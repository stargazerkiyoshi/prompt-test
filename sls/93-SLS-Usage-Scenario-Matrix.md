# SLS Usage Scenario Matrix

## 1. 文件定位

本文件用于整理 SLS 的使用场景，说明哪些任务适合使用 SLS、应使用哪一类模式、以及是否需要先列大纲或学习路径。

本文件属于使用设计 / 调用设计层，不属于 Core 层、结构层、编排层、叙述层的核心规范文件。

本文件不重新定义 `10-SLS-Core.md`、`20-SLS-Architecture.md`、`30-SLS-Teaching-Flow.md`、`40-SLS-Narrative-Guide.md`，也不替代 `90-SLS-Prompt-Assembly.md` 的调用组织职责，或 `92-SLS-Prompt-Profiles.md` 的可复制 prompt 资产职责。

本文件的作用是：

1. 统一整理 SLS 当前适用的任务场景。
2. 说明不同场景下的对象类型、目标与展开方式。
3. 帮助判断某个任务是否适合先列大纲、路径或比较维度。
4. 为后续扩展 `92-SLS-Prompt-Profiles.md` 提供上游依据。
5. 让“什么时候该用哪种模式”这件事更明确。

## 2. SLS 适用场景的判断原则

通常满足以下特征的任务，适合使用 SLS：

- 需要结构化展开，而不是一句定义或一句结论即可完成。
- 主题内部存在概念关系、步骤关系、层级关系或前后依赖。
- 若直接一口气回答，容易讲乱、讲散、讲成堆点摘要。
- 需要同时处理直觉、概念、机制、形式化或整体骨架中的至少两类内容。
- 需要控制叙述节奏与形式化密度，避免过早压缩或过早抽象。
- 任务目标不仅是“给答案”，而是“建立理解”。

通常不必优先使用 SLS，或只需轻量使用 SLS 的任务包括：

- 单一事实查询、术语释义、极短定义类问题。
- 只需要一句判断、一步操作或一个直接结论的问题。
- 局部补丁式答疑，且问题边界很窄，不涉及结构重建。
- 已经有清晰主线，只需补一处缺口，不需要重新铺设学习路径。

判断时可优先看三个问题：

1. 这个任务是否需要“展开”而不只是“回答”？
2. 这个任务是否需要“结构”而不只是“信息”？
3. 这个任务是否值得先给骨架，再进入具体讲解？

若三个问题里至少有两个答案是“是”，通常适合使用 SLS。

## 3. 使用场景总览

当前可归纳的主要使用场景如下：

1. 完整主题学习
2. 复杂单概念讲解
3. 简单单概念解释
4. 机制 / 流程 / 方法讲解
5. 关系辨析
6. 模型 / 系统 / 理论框架整体理解
7. 章节 / 模块预览
8. 复习 / 总结 / 回顾
9. 重讲 / 调节
10. 小范围补丁式答疑

这些场景可进一步理解为三类：

- 需要先建整体骨架的场景：完整主题学习、复杂单概念讲解、机制 / 流程 / 方法讲解、关系辨析、模型 / 系统 / 理论框架整体理解、章节 / 模块预览。
- 可按规模决定是否先给大纲的场景：复习 / 总结 / 回顾。
- 通常可直接进入讲解的场景：简单单概念解释、重讲 / 调节、小范围补丁式答疑。

## 4. 各场景的详细说明

### 4.1 完整主题学习

场景定义：
围绕一个相对完整的主题做系统学习，目标不是回答单题，而是建立逐步推进的整体理解。

对象特点：
主题范围较大，通常包含多个章节、多个核心概念和明显的前后依赖关系。

典型例子：
“系统学习微积分”“带我从头理解操作系统”“按章节学习贝叶斯统计”。

SLS 是否适用：
适用，且是高匹配场景。

是否建议先列大纲：
建议。应先给学习路径、章节结构或递进关系，再进入章节讲解。

适合的后续 profile 方向：
完整主题学习常规版、展开版、轻量版、路径先行版。

使用时的注意事项：
不要把“先列路径”误写成只给目录；路径之后应进入正式讲解。章节之间要有承接理由，不能只并列堆章节。

### 4.2 复杂单概念讲解

场景定义：
对象表面上是“一个概念”，但该概念内部关系复杂，无法靠一句定义讲清。

对象特点：
通常涉及问题背景、概念建立过程、相邻概念关系、必要形式化和例子。

典型例子：
“讲清楚梯度下降”“理解熵到底是什么”“解释 Kolmogorov complexity”。

SLS 是否适用：
适用，是单概念场景中的重点类型。

是否建议先列大纲：
建议。应先给概念学习大纲或讲解骨架，再进入分节讲解。

适合的后续 profile 方向：
单概念增强版、概念学习大纲版、带辨析版、例子驱动版。

使用时的注意事项：
不要被“单概念”三个字误导成短答。若对象本身有多层含义，应显式分层，不要直接抛最终定义。

### 4.3 简单单概念解释

场景定义：
需要把一个范围较小、边界较清楚的概念说明白。

对象特点：
通常不需要完整章节递进，但仍可能需要一个问题入口、一个例子和一句边界说明。

典型例子：
“什么是哈希表”“解释一下偏导数”“什么叫范数”。

SLS 是否适用：
适用，但一般使用轻量形态即可。

是否建议先列大纲：
一般不必。通常可直接讲解，必要时只给一个很短的讲解顺序提示。

适合的后续 profile 方向：
单概念常规版、简洁直讲版、轻量例子版。

使用时的注意事项：
不要过度套入完整主题模式。重点是清楚、自然、边界明确，而不是人为拉长结构。

### 4.4 机制 / 流程 / 方法讲解

场景定义：
对象重点不在静态定义，而在“它如何运作”“按什么步骤发生”“如何执行这个方法”。

对象特点：
具有顺序性、阶段性、输入输出关系或操作步骤。

典型例子：
“TCP 三次握手是怎么工作的”“快速排序的流程是什么”“如何做梯度反向传播”。

SLS 是否适用：
适用。

是否建议先列大纲：
建议。应先给流程骨架、阶段列表或步骤大纲，再逐步展开各步骤。

适合的后续 profile 方向：
流程骨架版、逐步展开版、方法教学版、机制图景版。

使用时的注意事项：
不要只列步骤名称；应说明每一步在解决什么问题、与前后步骤如何衔接、在哪里容易误解。

### 4.5 关系辨析

场景定义：
重点任务是区分几个相近对象，建立边界、联系和不可替代之处。

对象特点：
对象之间往往表面相似，用户通常“都看过定义，但仍会混”。

典型例子：
“区分相关与因果”“比较生成式 AI 与判别式 AI”“辨析充分条件、必要条件、充要条件”。

SLS 是否适用：
适用。

是否建议先列大纲：
建议。应先列比较对象与比较维度，再进入辨析讲解。

适合的后续 profile 方向：
强辨析版、比较维度版、边界澄清版、对比例子版。

使用时的注意事项：
不要把辨析写成并列定义汇总。重点是边界、分叉点、同一情境下的不同使用方式。

### 4.6 模型 / 系统 / 理论框架整体理解

场景定义：
对象是一个由多个组件、层次或子概念构成的整体，用户需要先看清整体框架，再进入局部。

对象特点：
强调整体结构、组件关系、信息流、约束关系或理论位置。

典型例子：
“Transformer 整体怎么理解”“数据库系统的整体结构”“博弈论有哪些核心框架”。

SLS 是否适用：
适用。

是否建议先列大纲：
建议。应先给整体框架或组件关系大纲，再按结构展开。

适合的后续 profile 方向：
系统框架版、组件关系版、整体到局部版、结构导览版。

使用时的注意事项：
不要一开始就钻进局部细节。若没有整体图景，后续细节容易失去位置感。

### 4.7 章节 / 模块预览

场景定义：
在正式深入之前，先预览某一章、某一模块或某一段学习路径。

对象特点：
范围比完整主题小，但仍有明确的内部结构和推进顺序。

典型例子：
“先预览一下线性代数里特征值这一章”“给我看一下这门课下一模块会讲什么”。

SLS 是否适用：
适用。

是否建议先列大纲：
建议。可视为小型路径模式，先给模块结构，再说明各部分承担什么作用。

适合的后续 profile 方向：
章节预览版、模块路径版、学习入口版。

使用时的注意事项：
预览的目标是建立进入感，不是提前把全部内容讲完。应控制深度，保留后续展开空间。

### 4.8 复习 / 总结 / 回顾

场景定义：
在已有一轮学习或已有基础理解后，重新压缩、重建、校准主题结构。

对象特点：
重点从“首次建立理解”转为“结构重建、主线回收、易混点回顾、可复述化”。

典型例子：
“帮我复习概率论主线”“把这部分知识重新总结成一套结构”“考试前快速回顾机器学习基础”。

SLS 是否适用：
适用。

是否建议先列大纲：
视规模而定。范围大时建议先列回顾结构；范围小时可直接压缩回顾。

适合的后续 profile 方向：
复习常规版、压缩回顾版、结构回顾版、易混点回顾版。

使用时的注意事项：
应强调压缩、结构重建和关键误解回收，而不是重新完整讲一遍。若用户只要唤回记忆，应控制展开密度。

### 4.9 重讲 / 调节

场景定义：
针对已经讲过的内容，按新的风格要求重新讲一次，或对展开度、自然度、密度进行调节。

对象特点：
核心内容通常不变，变化的是表达方式、展开程度和叙述节奏。

典型例子：
“刚才那段再讲得自然一点”“换成更展开的版本”“给我一个更轻量的解释”。

SLS 是否适用：
适用。

是否建议先列大纲：
一般不必。通常直接基于已讲内容做再调用更合适。

适合的后续 profile 方向：
重讲版、展开调节版、轻量重述版、自然化重述版。

使用时的注意事项：
重点是保持原任务不漂移，只调表达，不重开新主题。若用户只是要改口径，不应再次铺完整路径。

### 4.10 小范围补丁式答疑

场景定义：
用户已经有主线，只在一个局部点上存在小缺口，需要补一处定义、一步推理或一个局部例子。

对象特点：
范围窄、目标单一、上下文依赖强。

典型例子：
“这里为什么要取极限”“这一步公式怎么来的”“这个术语在刚才那段里是什么意思”。

SLS 是否适用：
部分适用。可作为后置或轻量场景处理。

是否建议先列大纲：
通常不必。

适合的后续 profile 方向：
局部答疑版、补丁解释版、上下文回扣版。

使用时的注意事项：
不要为了补一处缺口而重新展开完整体系。若局部问题暴露的是整体结构缺失，再升级为复杂单概念或复习类场景。

## 6. 场景优先级建议

### 6.1 第一优先级

- 完整主题学习
- 复杂单概念讲解
- 复习 / 总结 / 回顾

理由：

- 这三类最能代表 SLS 当前的核心价值，即结构化展开、节奏控制与理解建立。
- 现有 `92-SLS-Prompt-Profiles.md` 已经在这些方向上有基础，继续扩充的收益最高。
- 这三类场景覆盖面大，也最容易暴露调用层与展开密度的真实问题，适合作为 profile 体系的主干。

### 6.2 第二优先级

- 机制 / 流程 / 方法讲解
- 关系辨析

理由：

- 这两类场景很常见，也明显需要结构化，但其核心对象更具体，适合作为主干场景之后的专项增强。
- 它们对“先给骨架再展开”的需求非常稳定，适合沉淀成更专门的 profile 分支。

### 6.3 第三优先级

- 模型 / 系统 / 理论框架整体理解
- 章节 / 模块预览
- 重讲 / 调节
- 小范围补丁式答疑
- 简单单概念解释

理由：

- 这些场景重要，但要么与已有场景高度重叠，要么更偏轻量调节，不必在当前阶段优先拆出过多独立 profiles。
- 其中一部分可先用现有主 profile 加调节短句覆盖，等积累了稳定差异后再独立沉淀。
- 先保持 profile 体系主干清楚，比过早铺开大量细分模板更重要。

当前阶段更适合采用“先主干、后专项、再细分调节”的顺序。这样能避免 profile 过早碎片化，也更便于验证哪些场景差异值得独立成型。

## 7. 与现有文件的关系

与 `90-SLS-Prompt-Assembly.md` 的关系：

- `90` 负责说明如何按任务模式组织调用、组合文件与补充约束。
- 本文件负责说明有哪些场景、这些场景如何分类，以及它们是否适合先列大纲。
- 可先用本文件判断场景，再回到 `90` 选择调用模式与文件组合。

与 `92-SLS-Prompt-Profiles.md` 的关系：

- `92` 负责提供可直接复制的 prompt 模板。
- 本文件不提供具体 prompt 资产，而是为后续该补哪些 profile、哪些可先复用现有 profile 提供依据。
- 可把本文件视为 `92` 的上游分类面板，而不是模板仓库本身。

与未来 profile 扩展判断的关系：

- 当出现新任务类型时，可先看它是否能落入本文件已有场景。
- 若能落入已有场景，优先复用或微调现有 profile。
- 若反复出现且与已有场景存在稳定差异，再考虑为 `92` 新增独立 profile。

因此，本文件的维护目标不是收集所有任务，而是稳定维护一套“场景分类与适配判断”的公共基线。

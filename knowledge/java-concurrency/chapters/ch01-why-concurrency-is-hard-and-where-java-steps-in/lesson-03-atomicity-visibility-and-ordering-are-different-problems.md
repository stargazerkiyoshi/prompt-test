# 第 1 章 第 3 课：原子性、可见性、有序性为什么是三种不同麻烦

上一课我们已经把问题现场压缩到了一个很关键的位置：并发 bug 通常围绕共享可变状态出现，而且一旦结果依赖线程之间未经控制的时序，就会进入竞态条件。

但如果你再往前走一步，会发现“线程不安全”其实太笼统了。因为并发里的错误，至少会从三条不同路径冒出来：

1. 一段操作本来应该作为整体完成，却被别的线程插了进来。
2. 一个线程明明改了值，另一个线程却没有及时看见。
3. 代码看起来按这个顺序写，但别的线程观察到的效果却像换了顺序。

这三类问题，分别对应原子性（Atomicity）、可见性（Visibility）和有序性（Ordering）。

所以这节课的任务，是把“并发很乱”进一步拆清楚：

并发不是一种问题，而是三类不同缺口叠在一起。

## 先看原子性：为什么“一整步”其实可能会被拆开

原子性说的是：

一段操作要么整体完成，要么整体不发生，中间不会被别的线程看见半成品，也不会被别人插队破坏。

上一课的 `count++` 和库存扣减，其实就是典型的原子性问题。表面上看起来像一步，实际上却是读、算、写三个动作。一旦中间可以被其他线程插进来，就可能出现丢失更新。

所以原子性最先回答的是：

“这几步本来应该绑在一起的动作，能不能被拆开观察和打断？”

如果答案是“能”，那你就要警惕。

## 再看可见性：为什么值改了，别的线程却像没收到消息

很多人第一次学并发，会下意识觉得：

线程 A 把某个字段改了，线程 B 之后再读，当然应该读到新值。

这个直觉在单线程里成立，但在并发里并不总成立。因为线程对共享变量的读取和写入，不一定每次都直接以你脑中“全体线程共看一块板”的方式发生。

先看一个经典停止标志例子：

```java
public class StopDemo {
    private boolean stopped = false;

    public void stop() {
        stopped = true;
    }

    public void work() {
        while (!stopped) {
            // do work
        }
    }
}
```

如果一个线程执行 `work()`，另一个线程稍后执行 `stop()`，你可能会以为循环一定会停。但在缺少同步语义的情况下，工作线程未必会及时看到 `stopped = true` 这个更新。

这时候问题不是“写有没有发生”，而是“写发生后，别的线程什么时候能可靠地看见”。

这就是可见性在讨论的核心。

你可以先把它理解成：

共享变量的更新，不是天然会立刻、可靠、同步地广播给所有线程。

下一章我们会正式进入 JMM 解释它为什么如此，但现在先把现象认清。

## 再往前看有序性：源码顺序和观察顺序不一定完全一样

并发里第三类麻烦更隐蔽一些。有些时候，每个线程内部的代码看起来都按顺序写好了，可别的线程观察到的效果却像顺序被调换了一样。

先看一个很常见的初始化模式：

```java
resource = new Resource();
ready = true;
```

另一个线程可能这么读：

```java
if (ready) {
    use(resource);
}
```

如果没有足够同步约束，另一个线程未必总能稳定地观察到“只要 `ready == true`，那么 `resource` 一定已经是完整可用的”。也就是说，你在源码里写出来的先后关系，并不一定能无条件变成跨线程可观察的先后关系。

这就是有序性问题要处理的东西。它不是说 JVM 会随便把一切乱排，而是说：

在不破坏单线程语义的前提下，编译器、JIT 和处理器都可能对执行顺序做优化；而这些优化一旦进入多线程观察，就需要同步语义来约束。

## 为什么这三件事必须分开，而不能混成一句“线程不安全”

现在你已经能感觉到，这三类问题虽然都发生在并发场景里，但它们不是一个东西。

原子性关心的是“操作能不能被拆开和插队”。

可见性关心的是“一个线程的更新，另一个线程能不能及时可靠地看到”。

有序性关心的是“跨线程观察时，动作之间的先后关系有没有被正确建立”。

这三件事之所以要分开，是因为不同工具解决的缺口并不完全一样。

比如：

- `volatile` 主要补的是可见性，并建立一定的有序性约束，但不天然保证复合操作原子性。
- 锁通常既能提供互斥，也能在进入和退出临界区时建立可见性与顺序约束。
- 原子类能解决特定更新操作的原子性，但并不自动替你维护复杂状态不变量。

如果你不先把这三件事拆开，后面就很容易陷入“这个关键字到底是不是线程安全”的模糊问法。

更好的问法是：

当前这段代码缺的，到底是原子性、可见性，还是有序性？

## 用库存服务把三类问题同时放到一张图里

现在回到贯穿例子：电商库存与订单处理服务。

原子性问题会出现在什么地方？比如“检查库存是否大于 0，再扣减库存，再创建订单”这几个动作本来应该作为一个受控整体执行，否则就可能超卖。

可见性问题会出现在什么地方？比如管理员把某个商品标记为停售，后台线程已经写了 `stopped = true`，但工作线程还在继续处理下单请求，因为它没有及时看到这个更新。

有序性问题会出现在什么地方？比如一个线程先初始化订单上下文，再把“订单可处理”标志置为 true。另一个线程如果先看到了标志，却还没看到上下文完整初始化，就可能在一个半成品状态上继续工作。

这三个问题表面上都叫“并发 bug”，但真正修复时需要的工具和思路并不相同。

因此，到这里你应该开始形成一种更成熟的诊断习惯：

遇到并发问题时，不要先问“我要不要加锁”，而要先问“当前缺的是哪种保证”。

## 在必要处引入最少量形式化：为什么 Java 不能不谈内存模型

到目前为止，我们一直在讲直觉层。但现在已经可以引出一点最必要的形式化动机了。

一旦多个线程共同运行，程序正确性就不再只取决于“源码里写了什么”，还取决于：

1. 哪些读写动作对其他线程可见。
2. 哪些动作之间建立了先后关系。
3. 哪些复合操作必须被视为一个不可分割的单元。

这意味着 Java 不能只告诉你“语言里有 `if`、`for`、对象和方法”，它还必须告诉你：

在多线程场景里，什么样的读写关系是被保证的，什么样的先后关系可以被其他线程可靠观察。

这就是 Java 内存模型会在下一章出现的原因。它不是额外附赠的理论，而是 Java 对“跨线程语义”给出的统一约束。

## 把本课的完整例子走完：停止标志为什么足以暴露两类问题

我们把停止标志例子完整走一遍。

```java
public class InventoryWorker {
    private boolean stopped = false;

    public void stop() {
        stopped = true;
    }

    public void run() {
        while (!stopped) {
            processOrder();
        }
    }
}
```

如果你只用单线程直觉看，会觉得非常简单：管理线程调用 `stop()` 之后，`stopped` 就变成 true，工作线程下一次判断自然就退出。

但并发语境下，这段代码至少暴露两个问题。

第一，它缺可见性保证。工作线程未必会及时观察到 `stopped` 的最新值。

第二，它还在提醒你：程序正确性不仅仅是“字段最终值是什么”，还包括“这个值的改变有没有可靠地传到需要它的线程那里”。

而如果你把这个停止标志再和其他初始化状态拼在一起，例如：

```java
resource = loadResource();
stopped = false;
```

另一个线程再根据某个标志决定是否使用 `resource`，那你很快又会碰到有序性问题。

也就是说，一个很小的布尔标志，就足够让你看到：

并发世界里的麻烦不总来自复杂数据结构，很多时候来自很简单的字段缺少必要语义。

## 常见误解与当场修正

误解一：线程安全就是“结果不要错”，细分没必要。

修正：如果不区分原子性、可见性和有序性，你很难判断应该用什么工具，也很难解释为什么某个工具不能解决所有问题。

误解二：只要一个字段类型简单，比如 `boolean`，那更新应该天然安全。

修正：字段简单不等于跨线程语义完整。可见性和有序性问题照样可能出现。

误解三：只要用了 `volatile`，所有并发问题就解决了。

修正：`volatile` 很重要，但它不等于万能同步。它不能把所有复合操作自动变成原子操作。

## 小结：这节课到底拿到了什么

今天这节课，我们把“线程不安全”进一步拆成了三类不同问题：

原子性关注操作整体能不能被拆开，可见性关注线程之间能不能看见彼此更新，有序性关注跨线程观察到的先后关系能不能成立。

我们看到，库存扣减主要暴露原子性，停止标志暴露可见性，初始化与发布关系会把有序性也拉进来。到这里，你已经不应该再把并发问题看成一团模糊混乱，而要开始按缺口分类。

如果把本课压缩成一句高层直觉，那么就是：

并发的麻烦不是一个坑，而是三种不同的保证同时开始变得必要。

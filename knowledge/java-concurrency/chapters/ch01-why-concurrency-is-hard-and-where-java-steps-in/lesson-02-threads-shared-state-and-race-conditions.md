# 第 1 章 第 2 课：线程、共享状态与竞态条件

上一课我们先把入口问题看清了：并发最先打破的，是单线程里的顺序直觉。现在问题自然会更具体一点：

到底是什么东西，让多线程程序开始危险起来？

答案不是“线程很多”本身，而是：

多个线程开始共享同一份可变状态。

如果线程彼此完全独立，不共享数据，它们并发运行当然也有调度和资源问题，但不会立刻演化成线程安全问题。真正让程序结果变得不稳定的，是多个线程既能同时运行，又能读写同一份状态，而且这些读写之间没有受控关系。

所以这节课的任务，是把“并发会出错”进一步压缩成一个更明确的判断：

并发 bug 的核心现场，通常发生在共享状态被多个线程无序访问的时候。

## 先把三个对象分开：线程、状态、共享

要把并发讲清楚，先别急着上术语，先把三个对象分开。

第一，线程（Thread）是执行路径。你可以把它先理解成“程序里一条独立推进的工作线”。

第二，状态（State）是程序运行过程中会变化的内容，比如库存值、订单状态、缓存映射、任务队列长度。

第三，共享（Shared）意味着这些状态不只被一个线程独占使用，而是可能被多个线程同时读取、更新或依赖。

这三个对象单独看都不神秘，但它们一旦合在一起，问题就出来了。线程本来就有自己的执行节奏，状态本来就会变化，共享又让这些变化不再可由单一线程独占控制。于是程序的正确性开始依赖：

不同线程在什么时机接触同一份状态，以及它们之间有没有同步协议。

这时候你可以先记住一个非常实用的判断句：

并发本身不是风险，共享的可变状态才是风险放大器。

## 什么叫竞态条件：结果依赖谁先跑到哪一步

并发领域里有个核心词叫竞态条件（Race Condition）。

它说的不是“线程会竞争 CPU”这么泛，而是：

程序最终结果依赖多个线程谁先执行到哪一步，而这种先后顺序又没有被你明确控制。

这句话里最重要的是“结果依赖时序”。如果线程 A 先一步执行就得到结果 X，线程 B 先一步执行就得到结果 Y，而且这种差异不是你业务规则有意设计出来的，那就很可能已经进入竞态条件。

我们还是回到库存例子。问题不在于两个线程都能执行，而在于：

两个线程都在没有同步约束的情况下，基于同一份库存做检查和更新。

于是哪个线程先读、哪个线程后写、谁覆盖了谁，都会影响结果。这种“结果跟时机绑定”的现象，就是竞态条件最典型的形态。

## 为什么“先检查再执行”在并发里特别危险

很多业务代码都喜欢写成这样：

```java
if (stock > 0) {
    stock--;
}
```

在单线程里，这种“先检查再执行”的逻辑很自然。但在并发里，它会变成一个特别容易出问题的模式。

因为这里其实包含两个阶段：

1. 检查当前状态是否满足条件。
2. 基于刚才的检查结果执行更新。

只要这两个阶段之间不是不可分割的，别的线程就可能在中间把状态改掉。于是你在第 1 步得到的判断，到了第 2 步已经不再成立。

这类问题的本质并不是 if 语句有错，而是：

检查时看到的世界，和真正执行更新时的世界，可能已经不是同一个世界。

这一步非常重要，因为后面你会不断遇到这种模式：

- `if (!map.containsKey(k)) { map.put(k, v); }`
- `if (balance >= amount) { balance -= amount; }`
- `if (!stopped) { doWork(); }`

它们在单线程里都很顺，但在并发里都可能隐藏类似风险。

## 用一个计数器例子，把竞态条件看得更直接

我们看一个更小的例子。假设有一个共享计数器，两个线程各自执行一万次自增：

```java
public class CounterDemo {
    private int count = 0;

    public void increment() {
        count++;
    }

    public int getCount() {
        return count;
    }
}
```

如果线程 A 和线程 B 都执行：

```java
for (int i = 0; i < 10000; i++) {
    counter.increment();
}
```

很多人第一次看会直觉觉得，最后结果应该是 20000。可实际上，不加任何同步时，结果往往会小于 20000。

为什么？因为 `count++` 不是一个不可分割的整体。它通常可以理解成：

1. 读出 `count` 当前值。
2. 加 1。
3. 写回新值。

只要两个线程在这些步骤上发生交错，就可能都读到同一个旧值，然后其中一次更新被覆盖掉。

比如：

1. 线程 A 读到 `count == 5`。
2. 线程 B 也读到 `count == 5`。
3. 线程 A 写回 6。
4. 线程 B 也写回 6。

看起来两个线程都做了加 1，但全局结果只增加了 1。

这类现象有时也叫丢失更新（Lost Update）。它是竞态条件最经典的具体后果之一。

## 共享状态为什么会从“普通字段”变成“协作协议”

在单线程程序里，字段通常只是“保存值的地方”。但一旦字段被多个线程共享，它的地位就变了。

此时这个字段不再只是一个变量，而是多个线程彼此协作时必须共同遵守的一部分协议。

比如库存值不再只是“库存是多少”的记录，它还隐含着：

- 谁可以在什么条件下修改它；
- 修改时别人能不能同时读写；
- 读到它的人是否看到了最新值；
- 它的变化是否和其他字段变化保持一致。

也就是说，所谓线程安全，并不只是“最后值对不对”，而是“围绕共享状态的访问有没有受控规则”。

这时候你会发现，并发编程和普通面向对象设计有一个很大的差别：

对象封装只解决“谁能调用什么方法”，但不自动解决“多个线程同时调用时会发生什么”。

## 为什么局部变量通常比共享字段安全得多

这里顺手建立一个很实用的判断能力。

为什么很多并发问题都围绕字段、集合、缓存、连接池状态出现，而很少围绕方法内部临时变量出现？

因为方法内部的局部变量通常只被当前线程看见，不共享，就没有线程间争抢。

比如：

```java
public int sum(int a, int b) {
    int result = a + b;
    return result;
}
```

这里的 `result` 是当前调用栈里的临时值，每个线程各算各的，互不干扰。

但如果你把状态提升成实例字段，情况就变了：

```java
private int result;

public void sumAndStore(int a, int b) {
    result = a + b;
}
```

这时候多个线程就可能同时读写同一个 `result`。

所以“尽量减少共享可变状态”会成为并发设计里的高频原则，不是因为这句话听起来高级，而是因为它直接减少了竞态条件的发生面。

## 把库存例子再推进一步：竞态条件不只会超卖，还会破坏状态一致性

库存例子如果只停在“库存减错了”，还不够说明问题。真实系统里更危险的是：

共享状态常常不只一个字段，而是一组必须保持一致的字段。

比如订单系统里，你可能同时维护：

- `stock`
- `orderStatus`
- `reservedCount`
- `paymentTimeout`

如果多个线程分别在不同时间更新这些字段，但没有同步规则，就可能出现局部上看每一项都“改过了”，全局上看整个对象却进入不一致状态。

例如库存已经扣减，但订单状态还没更新；或者订单已经标记成功，但异步通知线程还看见旧库存。

这说明竞态条件不只会造成“数值不准”，还会造成“多个相关状态之间失去一致性”。后面讲锁和内存模型时，你会反复看到这一点。

## 常见误解与当场修正

误解一：只要线程不多，问题应该就不大。

修正：竞态条件和线程数当然有关，但它的本质不是“线程很多”，而是“有没有无序共享状态”。两个线程就足够把问题引爆。

误解二：只要字段类型是 `int`、`boolean` 这种简单类型，就不会复杂。

修正：类型简单不等于访问安全。并发问题常常不是因为数据结构复杂，而是因为读写关系没有被约束。

误解三：对象已经封装得很好了，所以应该线程安全。

修正：封装只说明访问入口被限制，不说明多个线程同时进入这些入口时会自动安全。

## 小结：这节课到底拿到了什么

今天这节课，我们把“并发为什么危险”进一步压缩成了一个更明确的结构：

线程安全问题往往出在共享的可变状态上，而竞态条件的核心特征，是程序结果依赖线程之间未经控制的执行时序。

我们看到，“先检查再执行”在并发里特别容易出错；`count++` 这种看起来很小的更新也可能丢失；共享字段一旦进入多线程语境，就不再只是普通变量，而变成了线程协作协议的一部分。

如果把本课压缩成一句高层直觉，那么就是：

多线程真正危险的，不是线程并行本身，而是它们在没有规则的情况下共同碰同一份状态。

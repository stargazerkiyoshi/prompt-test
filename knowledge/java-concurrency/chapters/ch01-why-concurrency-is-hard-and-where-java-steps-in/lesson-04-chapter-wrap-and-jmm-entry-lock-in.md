# 第 1 章 第 4 课：本章回收与 JMM 入口锁定

到这一课，我们先不再引入新的并发工具，而是把第 1 章真正搭出来的主线收紧。因为如果这一章只是让你记住几个名词，后面进入 `volatile`、锁和线程池时，你仍然可能把它们学成一组分散技巧；而如果这条主线被锁住，后面每出现一个工具，你都会知道它是在补哪一个缺口。

所以这一课要做两件事。第一，把前面三课的推进链完整合起来。第二，把这个链条稳稳交给下一章，让“Java 内存模型为什么必要”成为顺理成章的下一步，而不是课程突然拐弯。

## 先回到最开头的问题：我们到底在试图解决什么

第 1 章最开始问的，不是“Java 里有哪些并发 API”，而是一个更早的问题：

为什么多线程会让本来很普通的程序开始失控？

现在我们已经可以用更清楚的话来回答了。

并发之所以难，不是因为线程这个概念本身神秘，而是因为多个线程一旦共享可变状态，单线程里的顺序推理链就会断掉。程序结果开始依赖线程如何交错执行，状态检查和状态更新可能被别的线程插队打断，一个线程的修改也不一定会被另一个线程及时可靠地看到。再往前一步，即使每个线程内部代码都写得很顺，跨线程观察到的先后关系也不一定天然成立。

这句话就是第 1 章真正打下的地基。

如果这层地基不立住，后面你看到 `volatile` 时就可能把它理解成“一个奇怪关键字”；看到锁时就可能只觉得“这是一种加门的办法”；看到线程池时又会把它当成另一类无关工具。但如果这层地基立住了，后面你就会知道它们都在处理同一类结构问题：

如何让共享状态下的线程协作变得可控。

## 把前面三课压成一条连续链

现在我们把前三课的逻辑串起来，而且不是并列回顾，而是要看它们怎样一步推一步。

第一课先做了一件必须先做的事：把单线程直觉和多线程现实分开。我们看到，源码顺序不再是唯一重要顺序，一行代码也不一定是不可分割的一步，共享状态会让程序结果依赖执行交错。

第二课把问题进一步压缩到了线程、共享状态和竞态条件。我们看到，真正危险的不是线程数量本身，而是多个线程在没有同步规则的情况下共同访问同一份可变状态。于是“先检查再执行”“自增自减”“多字段一致性”都会变成风险源。

第三课则继续把这些风险细分成三种不同缺口：原子性、可见性和有序性。到这里，你已经不应该再用一句模糊的“线程不安全”来概括全部现象，而要开始按缺什么保证来诊断问题。

所以前三课连起来，不是“并发会出错”“共享状态有风险”“有三个术语”这几个散点，而是一条闭合推进链：

多线程会打破单线程推理；
风险集中出现在共享可变状态上；
共享状态的问题又会分别表现为原子性、可见性和有序性缺口；
于是 Java 必须提供一套跨线程语义，明确什么样的读写和先后关系是可靠的。

到这里，下一章的入口其实已经锁定了。

## 为什么说 JMM 的入口已经锁定，而不是只是“理论上值得了解”

这里有个很重要的区别。

如果你只是觉得“Java 内存模型可能有点重要”，那它看起来就像一段偏理论的额外知识。你后面会不停问：我写业务代码为什么非得懂这个？

但现在我们的结论已经更强了：

只要你承认并发 bug 不只是“代码写错”，而是线程间可见性、时序和共享状态语义的问题，那么 JMM 就不是可选背景知识，而是 Java 对这些问题给出的统一回答框架。

它的地位不是“把事情讲复杂”，而是“把原本到处散落的现象压成一组明确规则”。

也就是说，下一章我们不是为了学一个新概念而去学 JMM，而是因为前面的问题已经逼到这里：

如果不建立统一的内存与顺序语义，Java 并发就只能停留在经验猜测层。

## 把贯穿例子也收一遍：库存服务为什么足以说明问题

我们这一章一直在回收电商库存与订单处理服务。

这个例子之所以值得反复使用，不是因为它贴近面试题，而是因为它足够短，却已经把并发主线暴露出来了。

它先让我们看到：两个线程基于同一份库存做判断和更新时，执行交错会直接改变结果。

它又让我们看到：同样是共享状态问题，既可能表现为超卖这种原子性缺口，也可能表现为停售开关、订单状态发布这种可见性和有序性缺口。

它还提醒我们：真实系统里共享的常常不是一个简单变量，而是一组必须保持一致的状态。

于是这个例子几乎把本章全部关键矛盾都放到了台面上：

同一个系统里，多个线程既要共享状态，又要看到彼此的结果，还要按可靠顺序协作；如果缺少统一语义约束，程序就会在“偶现”和“看起来不该出错”的地方不断失控。

这正是下一章必须回答的问题。

## 这一章到底建立了哪几层能力

如果不按“知识点”列，而按真正的理解能力来讲，第 1 章至少帮你建立了四层东西。

第一层，是视角切换能力。你现在应该能识别：并发程序不能只按单线程方式阅读，执行交错本身就是程序语义的一部分。

第二层，是风险定位能力。你能看出共享可变状态是线程安全问题的主要入口，而不是把所有并发都模糊地归结为“线程太多”。

第三层，是问题分层能力。你能把麻烦分成原子性、可见性和有序性，而不是把所有问题混成一句“线程不安全”。

第四层，是动机判断能力。你不需要别人先告诉你“JMM 很重要”，也能顺着问题本身走到“Java 必须给出统一跨线程语义”这个结论。

请注意，这四层能力比“记住几个定义”重要得多。因为后一种只是局部知识，前一种才是真正能支撑你继续往下学的主干。

## 为什么我们这一章没有急着讲 `volatile`

到这里，有人可能会觉得：这一章是不是现象讲得多，关键字讲得少？

这其实是故意的，而且符合 SLS 的推进节奏。因为这一章的任务不是马上告诉你 `volatile` 怎么写，而是先让你知道为什么会需要它。如果动机没立住，后面的关键字和语义就会变成零散结论；如果动机立住了，后面每一条规则都会像在回答已经成熟的问题。

换句话说，这一章是在铺轨，而不是冲工具箱。它的价值不在于你已经掌握了 Java 并发全部机制，而在于你已经知道后面的机制要服务什么问题。

## 从本章自然迈向下一章：现在真正该问什么

到这里，下一章的问题已经不是泛泛的“请介绍一下 Java 内存模型”，而应该是一个非常具体的问题：

如果多个线程共享状态，而程序正确性又依赖可见性和先后关系，那么 Java 到底怎样规定这些读写何时对别人可见、哪些先后关系可以被可靠观察、哪些语义必须靠同步手段建立？

你看，问题已经成熟到可以进入机制层了。

我们不再只是知道“并发会乱”，而是已经把下一章的结构任务明确出来了：

1. 线程之间到底共享的是什么语义，而不只是字段值。
2. `volatile` 为什么能补可见性，但又为什么不是万能同步方案。
3. `happens-before` 为什么会成为理解并发顺序关系的核心语言。

这三步一旦被写清楚，Java 并发的第一层骨架就真正立起来了。

## 本章最终收束

如果把第 1 章整章压缩成一句高层直觉，那么就是：

Java 并发编程的出发点，不是“让代码同时跑起来”，而是“让共享状态下的线程协作有可靠语义”。

如果再把这句话压得更实用一点，那么你可以这样记：

前面这一章解决的不是 Java 并发怎么写，而是 Java 并发为什么必须有一套比单线程更严格的规则。

到这里，第 1 章的局部闭环已经完成。下一章开始，我们就不再停留在“为什么会乱”，而会正式进入“Java 到底怎样约束这种混乱”。

# 第 1 章 第 1 课：为什么单线程直觉一进多线程就会失灵

## 先从一个很常见的误会开始

如果你已经写过一段时间 Java 业务代码，很容易先形成一种默认直觉：

代码就是按顺序一行一行执行的。只要每一行自己没错，整段逻辑大概率也不会太离谱。

这个直觉在单线程世界里通常够用。你可以从上往下读代码，判断变量怎样变化、分支怎样选择、方法怎样调用，最后得到一个比较稳定的结果预期。

但并发编程最危险的地方就在这里：

你带着单线程里形成的正确阅读习惯进入多线程环境时，往往会在最开始就误判问题边界。

因为多线程不是“同一段顺序逻辑复制几份同时跑”这么简单。一旦多个线程同时存在，而且它们之间还会共享状态、争抢资源、彼此协调，程序结果就不再只由源码顺序决定，而会受到调度时机、执行交错和内存可见性的共同影响。

所以今天这节课的第一任务，不是立刻讲工具，而是先把一个入口问题讲透：

为什么单线程里正确的阅读方式，到了并发场景里会失效？

只有这个问题真的站稳，后面你才会明白，Java 并发工具的存在不是“语法变多了”，而是在补单线程直觉覆盖不到的缺口。

## 先看一个最朴素的问题：库存只剩 1，为什么还能卖出 2

我们用整门课的核心例子做入口。

假设系统里有一个库存字段：

```java
private int stock = 1;
```

扣减库存的代码看起来也很普通：

```java
if (stock > 0) {
    stock--;
    createOrder();
}
```

如果你站在单线程视角读这段代码，会很自然地觉得：

只要 `stock` 一开始是 1，第一个请求进来后会把它减成 0，后面的请求就进不去了。

这个判断在单线程里没问题，但一旦两个线程几乎同时执行，它就不再可靠。因为线程 A 可能先读到 `stock == 1`，线程 B 也可能在 A 还没来得及写回之前读到同样的值。于是两个线程都通过判断，都执行扣减，都创建订单。

这就是并发里最常见的一类现象：

源码没有变，单行语句也没有变，但多个线程的执行交错改变了结果。

你现在还不需要知道锁、CAS 或 JMM 细节，只要先把一个判断立住：

并发程序的结果，往往不只取决于“写了什么”，还取决于“谁先做、谁后做、谁和谁交错在了一起”。

## 为什么“源码顺序”不等于“运行时顺序”

很多初学者第一次接触并发时，会下意识地把源码顺序当成唯一顺序。比如看到：

```java
stock--;
createOrder();
```

就会觉得这两步天然不可分割，好像程序一定是“先扣库存，再建订单”，而且别的线程不会在中间插进来。

问题是，在多线程场景里，“当前线程内部的顺序”和“整个系统外部观察到的顺序”不是一回事。

线程 A 可以执行到一半被切走，线程 B 可以在这时运行；线程 B 也可以先完成自己的读写，再把控制权交回给 A。对最终结果来说，真正重要的不是某一段代码在一个线程里怎样排，而是多个线程的动作怎样交错。

这时候程序已经不像一条直线，更像多条线交织在一起。你想理解结果，不能只盯着某一条线往下看，而要开始考虑：

1. 哪些状态是多个线程共享的。
2. 哪些步骤表面上是一行，实际上可能包含多个子动作。
3. 哪些时机一旦交错，就会产生你原本没考虑到的分支。

这就是为什么并发编程的第一步不是背 API，而是重建阅读程序的视角。

## 从“代码本身没问题”到“交错之后就出问题”

我们把刚才的库存逻辑再拆得细一点。

表面上你只写了：

```java
stock--;
```

但从计算动作看，这通常至少包含三步意思：

1. 读取当前 `stock` 的值。
2. 在当前线程里计算 `stock - 1`。
3. 把新值写回去。

一旦你把这三步拆开，就很容易看见问题了。线程 A 和线程 B 都可能先后读取到同样的旧值，然后各自算出新值，再各自写回。于是某一次更新就被覆盖了。

这一步非常关键，因为它会打破一个常见误解：

“代码只有一行”不等于“操作就是原子的”。

在单线程里，你几乎不需要这样拆看一行代码；但在并发里，如果你不学会把表面上的一步展开成底层动作，你就很难准确判断线程安全。

## 并发真正改变的，不只是执行时机，还有你对状态的理解方式

在单线程程序里，状态变化通常是沿着一条清晰路径前进的。某个变量改了，你基本能顺着调用栈回头找到是谁改的、为什么改的。

但在并发程序里，状态不再属于某一条执行路径，而可能同时被多个执行路径观察、修改、依赖。于是“状态”这个概念本身就变了：

它不再只是“当前这段代码手里的一个变量”，而是“多个线程可能同时接触的一块共享现实”。

这就像你从一个人在白板上写字，变成多个人围着同一块白板同时写、擦、改。白板还是那块白板，但问题已经完全不一样了。你现在关心的不只是写字内容对不对，还关心谁什么时候写、谁看见了谁写的内容、有没有人把别人的更新覆盖掉。

所以并发编程的困难，并不只是“线程多了”，而是共享状态让程序从局部线性逻辑变成了全局协作问题。

## 为什么“偶现”会成为并发 bug 的标志性体验

并发 bug 还有一个特别讨厌的地方：它常常不是每次都出现。

同样一段代码，有时候压测会出错，有时候本地跑几百次又像没事；线上流量一上来出现一次超卖，日志一查却很难复现。

这不是因为 bug 不真实，而是因为它往往依赖特定时机和特定交错。只要线程切换点稍微不同，结果就可能不一样。

于是并发 bug 经常带来一种很误导人的感受：

“这代码大部分时间都对，所以应该问题不大。”

恰恰相反。并发领域里，“偶现”往往说明你已经踩到了时序相关问题，只是它还没有在每次执行里都暴露出来。

这也是为什么并发不能靠经验猜测。你不能因为某段代码线上一周只错一次，就把它归类为“小概率异常”。在共享状态和执行交错面前，小概率常常只是还没碰到足够坏的调度。

## 把核心例子在本课内完整走一遍

现在我们把库存例子从头走一遍，不只是点到为止。

假设初始库存为 1，两个请求线程 A 和 B 几乎同时进入：

```java
if (stock > 0) {
    stock--;
    createOrder();
}
```

一种可能的执行交错是：

1. 线程 A 读取 `stock`，看到值是 1。
2. 线程 A 还没写回，线程切换。
3. 线程 B 读取 `stock`，也看到值是 1。
4. 线程 B 扣减库存，写回 0，并创建订单。
5. 线程切回 A。
6. 线程 A 继续执行，基于自己刚才读到的旧值继续扣减，并再次创建订单。

请注意，这里没有任何一行源码“自己写错了”。真正出问题的是：

两个线程都基于过时但当时看起来合法的状态做了后续决策。

这就是并发里极其核心的一件事：

线程的判断和动作之间，如果没有受控同步，别的线程完全可能在你不注意的时候改变了世界。

因此，本课真正建立起来的不是一句“超卖是因为并发”，而是更精确的判断：

多线程程序里，状态检查和状态更新一旦可以被别的线程插队，单线程里的推理链就会断掉。

## 常见误解与当场修正

误解一：只要业务逻辑简单，并发也不会太复杂。

修正：并发复杂度很多时候不来自业务规则本身，而来自多个线程怎样共享和更新状态。即使业务逻辑只有两三行，只要存在共享状态，也可能不安全。

误解二：代码能跑通很多次，说明大概率没问题。

修正：并发 bug 的典型特征就是依赖时机，偶现不代表轻微，反而常常说明问题更隐蔽。

误解三：一行代码天然比多行代码更安全。

修正：在并发语境里，你必须区分“源码看起来是一行”和“底层动作是否不可分割”。这两者不是一回事。

## 小结：这节课到底拿到了什么

今天这节课，我们还没有进入 Java 的并发语义细节，而是先把一个入口判断立住了：

并发会让你在单线程里形成的源码顺序直觉失效。

我们看到，库存超卖这种问题并不是因为代码字面上写错，而是因为多个线程的动作发生交错；一行更新动作也可能包含读取、计算、写回多个子步骤；共享状态一旦进入，程序结果就会同时受到执行时机和线程协作方式影响。

如果把本课压缩成一句高层直觉，那么就是：

并发程序最先打破的，不是语法规则，而是你对“代码按顺序可靠执行”的默认想象。

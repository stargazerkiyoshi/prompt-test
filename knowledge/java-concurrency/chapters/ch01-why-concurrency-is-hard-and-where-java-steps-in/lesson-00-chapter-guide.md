# 第 1 章 第 0 课：本章导学与主线说明

这一章的任务，不是让你一上来就背 `Thread`、`volatile`、`synchronized` 的定义，而是先把一个更根本的判断能力建立起来：为什么一段在单线程里完全正常的 Java 代码，一旦放到多线程里，就可能出现你事后怎么读都觉得“不应该”的结果？

如果这个问题本身没有看清，后面你看到锁、原子类、线程池时，只会觉得它们是一些分散工具，而不知道它们为什么必须存在。

这一章要先把三层事实讲清楚。第一，多线程不是“代码复制几份同时跑”这么简单，一旦共享状态进入，就会出现竞争。第二，并发 bug 不只是一种，它至少会从原子性、可见性和有序性三个方向冒出来。第三，Java 之所以需要内存模型和同步语义，不是为了把编程变复杂，而是为了给这些混乱建立统一约束。

所以第 1 章在做的事，可以压缩成一句话：

先把“并发为什么会把程序搞乱”讲透，再让“Java 为什么必须有 JMM 和同步机制”变成自然结论。

读完这一章之后，你至少应该建立起三种稳定能力。

1. 能说明为什么“这段代码看起来只有一行更新”不等于多线程下安全。
2. 能区分原子性、可见性和有序性这三类不同困难。
3. 能用自己的话解释：Java 的并发工具不是凭空堆出来的，而是在回应共享状态下的结构性问题。

为了让后续章节不断回收同一条主线，这一章固定了一个贯穿例子：电商库存与订单处理服务。

其中最先出现的入口问题是：

“库存明明只有 1 件，为什么两个并发请求还能都下单成功？”

这个例子看起来很常见，但它足够暴露很多关键困难。你不能只看 `stock--` 这一行长什么样，还必须知道两个线程什么时候读、什么时候写、是不是看到了彼此的更新、以及程序执行顺序是不是和源码顺序完全一致。后面的章节里，这个例子会被不断回收。第 2 章会用它说明停售标志为什么可能不可见，第 3 章会用它展开互斥与条件等待，第 4 章会用它讲订单任务编排，第 5 章会把它带进队列消费，第 6 章会把它带进限流与原子更新。

## 学习提醒

这一章里如果你某一步觉得“这些现象我好像都听说过，但还是连不起来”，先回到两个问题：

- 当前麻烦来自“多个线程抢同一份状态”，还是来自“线程之间看见的状态不一致”？
- 当前问题是在说“这一小段操作不能被拆开”，还是在说“值改了但别人没看见”，还是在说“执行顺序和你想的不一样”？

只要这两个问题不丢，你到第 2 章进入 Java 内存模型时，就不会把它误解成纯理论层的额外负担。

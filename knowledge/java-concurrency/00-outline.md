# Java 并发编程系统学习路径（SLS 完整主题学习 + 展开讲解模式）

这份文档是第一阶段产物：先把“Java 并发编程”应该怎么学、为什么按这个顺序学、每一章在解决什么问题、以及章节之间怎样承接，完整搭出一张可执行的学习路线图。这里先不急着一口气把所有 API 和源码细节都铺开，而是先把整条主线立住。这样你后面进入任何一章时，都知道自己当前在解决哪一类并发问题，也知道这一章为什么必要。

这套路线按“已经会写 Java 业务代码，但并发知识零散或不稳定”的读者来设计。也就是说，我们默认你知道类、对象、方法、异常、集合、泛型这些基础内容，但不假定你已经真正理解线程安全、内存可见性、锁、线程池和 JUC 工具之间的关系。

整门课围绕一个总问题展开：

当多个线程同时操作同一份状态，而且这些线程还要在正确性、吞吐、延迟和资源成本之间平衡时，Java 到底提供了哪些机制来保证程序“既不乱、也不慢到不可用”？

如果把并发编程误解成“会开线程”或者“会用线程池”，你很快就会在真实系统里撞墙。因为并发真正难的地方，不是如何让两个线程同时跑起来，而是：

1. 共享状态为什么会变成风险源。
2. 线程之间为什么会彼此看不见、看不全、看得不一致。
3. 正确性为什么不能只靠“感觉上应该没事”。
4. Java 为什么需要从 `synchronized`、`volatile`、`Lock`、原子类、线程池、阻塞队列一路铺到完整的并发工具箱。

因此，这条学习路径会经历五次明显升级。第一步，先把“并发到底难在哪”看清，不再用单线程直觉判断多线程程序。第二步，进入 Java 内存模型，理解原子性、可见性和有序性为什么是三件不同的事。第三步，学习互斥、同步和协调，让线程之间能安全地共享状态。第四步，进入线程池、任务编排和并发容器，理解工程里“并发不是裸线程乱跑，而是受控执行”。第五步，回到更高层的设计、性能与诊断，知道什么时候该用什么工具，以及出了问题怎样定位。

## 章节递进总图：这门课为什么要这样排

第 1 章先解决“为什么并发会打破你在单线程里形成的直觉”这个入口问题。如果这一步没站稳，后面你看到 `volatile`、锁和线程池时，只会把它们当成分散的语法点，而不是在回应同一类结构性困难。

第 2 章承接第 1 章，进入 Java 内存模型（Java Memory Model, JMM）。这章会把“线程为什么会读到旧值、写丢更新、执行顺序看起来不合理”压缩成一套统一视角。也就是说，第 1 章让你看到问题，第 2 章开始解释问题为什么会出现。

第 3 章正式进入互斥与同步，理解 `synchronized`、`Lock`、`ReentrantLock`、`Condition` 和 `happens-before` 到底分别在补什么缺口。这一章解决的是：既然共享状态会出问题，Java 怎样提供受控访问方式。

第 4 章把视角从“共享状态保护”推进到“任务怎样被组织和调度”。你会看到为什么真实工程里不该手工乱建线程，为什么线程池、`Future`、`Callable`、`CompletableFuture` 会变成默认工具。

第 5 章进入并发容器与生产者-消费者模型。这里的重点不再是“一个变量怎么保护”，而是“多个线程怎样围绕一批数据结构安全协作”。`ConcurrentHashMap`、`CopyOnWriteArrayList`、`BlockingQueue` 都会在这个位置进入。

第 6 章继续进入更细粒度、更高性能的并发工具：CAS、原子类、`CountDownLatch`、`CyclicBarrier`、`Semaphore` 等。到这里你会第一次比较系统地看到：不是所有同步都靠一把大锁，很多场景更适合协调原语或无锁思路。

第 7 章做全局回收，回答“工具会用了以后，怎样设计、怎样调优、怎样排障”。这章会讨论线程池参数、锁竞争、死锁、饥饿、活锁、上下文切换、可观测性和并发代码审查要点，让课程最后回到工程落地。

## 贯穿全课程的核心例子

为了避免每一章都像孤立知识块，整门课固定一个主例子：电商库存与订单处理服务。

为什么选它？因为它几乎天然包含并发编程的主要矛盾。

第一，多个请求会同时扣减同一件商品库存，这会暴露竞态条件与原子性问题。

第二，请求线程、支付回调线程、异步通知线程可能同时读写订单状态，这会暴露可见性和有序性问题。

第三，库存校验、订单创建、支付确认、消息投递并不是同一种任务，它们会把线程池、阻塞队列和协调原语都拉进来。

第四，一旦流量上来，锁该加在哪、线程池该怎么配、热点数据结构该怎么选，会直接影响吞吐和延迟。

这个例子会在不同章节里被不断升级解释。第 1 章它只是“为什么超卖会出现”的入口问题；第 2 章它会用来说明旧值、不可见和重排序；第 3 章它会进入锁与条件队列；第 4 章它会变成任务编排与异步执行；第 5 章它会进入订单队列和并发容器；第 6 章它会进入限流、栅栏和原子更新；第 7 章它则会帮助我们讨论热点锁、线程池堆积和诊断方法。

## 分章与课时安排（先给路径，不在本文件里展开全部正文）

### 第 1 章 并发为什么难：单线程直觉在这里失效

这一章先不急着讲工具，而是先建立问题感。你会看到：在单线程里“这段代码看起来没问题”，并不意味着在多线程里也没问题。多个线程并发执行时，读写交错、执行时机和 CPU/缓存行为会一起改变程序结果。

课时安排：第 1 课“为什么单线程直觉一进多线程就会失灵”；第 2 课“线程、共享状态与竞态条件”；第 3 课“原子性、可见性、有序性为什么是三种不同麻烦”；第 4 课“本章回收：为什么 Java 必须引入内存模型与同步机制”。

### 第 2 章 Java 内存模型：`volatile`、可见性与重排序

这一章的任务是把第 1 章看到的混乱压缩成一个统一解释框架。为什么一个线程明明改了值，另一个线程却像没看见？为什么有些代码顺序改写后仍然“语义等价”，但多线程下却会出问题？这章会带你真正理解 JMM 在约束什么。

课时安排：第 1 课“主内存、工作内存与可见性直觉”；第 2 课“`volatile` 到底保证什么，不保证什么”；第 3 课“重排序与 `happens-before` 的入口”；第 4 课“用库存开关与停止标志走完一个完整例子”。

### 第 3 章 互斥与同步：锁不只是‘加一把门’

这章解决的是：共享状态既然危险，我们怎样受控访问它。你会从 `synchronized` 进入，再走到 `ReentrantLock`、`ReadWriteLock`、`Condition`，理解不同锁模型的能力边界和代价。

课时安排：第 1 课“为什么原子性常常需要互斥”；第 2 课“`synchronized` 的语义与适用边界”；第 3 课“`Lock` 与 `Condition` 在补什么”；第 4 课“本章回收：从库存扣减到等待通知模型”。

### 第 4 章 线程不是业务模型：线程池、`Future` 与任务编排

这章先解决一个工程误解：并发不是“需要异步就 new 一个线程”。你会看到线程资源为什么必须受控，为什么线程池是任务调度模型的一部分，为什么 `CompletableFuture` 适合表达任务依赖与组合。

课时安排：第 1 课“线程创建成本与线程池的必要性”；第 2 课“`ExecutorService`、`Callable`、`Future` 的任务模型”；第 3 课“`CompletableFuture` 如何表达并发编排”；第 4 课“本章回收：订单流水线怎样拆成受控异步任务”。

### 第 5 章 并发容器与队列协作：让数据结构也进入并发语境

当并发不再只是多个线程抢一个变量，而是多个线程共同消费和维护一批共享数据时，容器的选择会直接影响正确性和性能。这章会进入 `ConcurrentHashMap`、`CopyOnWriteArrayList`、`BlockingQueue` 和生产者-消费者模型。

课时安排：第 1 课“为什么普通集合在并发下会出问题”；第 2 课“`ConcurrentHashMap` 与读写并发直觉”；第 3 课“`BlockingQueue` 与生产者-消费者”；第 4 课“本章回收：订单事件流怎样进队列、出队列、被消费”。

### 第 6 章 更细粒度的并发工具：CAS、原子类与协调原语

到这里我们开始比较不同同步思路。锁当然重要，但并不是所有问题都该上锁。计数、状态位、限流、分阶段等待、线程会合，这些场景更适合原子类或协调原语。

课时安排：第 1 课“CAS 与原子类：什么时候不用锁也能安全更新”；第 2 课“`CountDownLatch`、`CyclicBarrier`、`Semaphore` 各管什么”；第 3 课“从库存闸门到批处理会合：协调原语的完整例子”；第 4 课“本章回收：工具选择不是越高级越好，而是匹配问题”。

### 第 7 章 设计、调优与排障：并发代码写出来以后怎么办

最后一章不再新增很多底层工具，而是做工程收束。你会讨论线程池参数、队列堆积、锁竞争、死锁和可观测性，理解并发代码不是“写完能跑”就结束，而是必须能审、能测、能诊断。

课时安排：第 1 课“线程池、队列与吞吐-延迟权衡”；第 2 课“死锁、饥饿、活锁与常见排障思路”；第 3 课“并发代码审查与测试策略”；第 4 课“总复盘：把整门课压回一张工具与问题映射图”。

## 整条学习主线再压缩一遍

如果把这门课压缩成一句最核心的话，那么主线就是：

Java 并发编程的本质，不是学会多少个类，而是学会怎样在多线程共享状态下同时维护正确性、协作性和成本边界。

如果再换成学习动作来表述，那么这条路线是：

先看见并发如何打破单线程直觉，再理解 Java 为什么需要内存模型；先掌握互斥与可见性，再进入任务调度、并发容器和协调原语；最后回到工程设计与诊断，把工具真正放回问题场景。

# 第 3 章 第 1 课：为什么它叫 Self-Attention

到这一章，我们终于进入 Transformer 的核心发动机了。但在一上来写公式之前，有一个问题必须先讲清楚，不然后面你很容易把 self-attention 学成一个流程名词：

为什么它叫 self-attention？

这个问题看似只是命名问题，实际上牵连着整个机制的对象关系。如果这个“self”没理解清楚，后面的 query、key、value 虽然也能跟着公式走，但你会搞不清模型到底是在“查自己”，还是“查别的东西”，更搞不清它和后面 cross-attention 的差别。

所以这一课的任务，是先把 self-attention 的对象关系彻底厘清。

## 先回到第 2 章的机制骨架：当前位置会去检查其他位置

上一章我们已经知道，注意力的基本动作链是这样的：

当前位置提出 query，其他位置提供 key 和 value，系统根据 query 和 key 的匹配强弱，决定各个 value 应该以多大权重参与当前位置表示的更新。

这套机制本身并没有规定 query、key、value 一定来自哪里。它只规定了角色分工。

也正因为如此，接下来最自然的问题就是：

在 Transformer 的 self-attention 里，当前位置要访问的“其他位置”究竟来自哪里？

答案就是：它们通常来自同一条序列。

所谓 self-attention，不是说一个位置只盯着自己看，也不是说模型在做某种“自我反思”。这里的 self 指的是：

一条序列内部的各个位置，彼此之间做 attention。

这句话很朴素，但非常关键。因为它意味着，模型处理某个位置时，它访问的信息源，不是另一条外部序列，也不是一个单独数据库，而就是当前这条输入本身。

## “self”不是只看自己，而是“在自己这条序列内部看别人”

这是初学者最容易误会的点之一。

当你第一次听到 self-attention，很容易从字面上理解成“自己注意自己”。如果真是这样，那机制好像就很无聊，因为当前位置不是本来就已经有自己的表示了吗？

但这里的 self，不是“只看当前位置自己”，而是“所有 query、key、value 都来自同一套输入表示”。

也就是说，假设当前句子有很多位置。每个位置都会先有一个初始表示，然后从这些表示里分别生成 query、key、value。接下来，每个位置都会拿着自己的 query，去和整句中所有位置的 key 做匹配，再把整句中各位置的 value 按不同权重汇总回来。

所以 self-attention 真正描述的是：

同一条序列内部，位置和位置之间彼此建立相关性并重写各自表示。

这个机制之所以厉害，就厉害在这里。因为它意味着，句子里任何一个位置，都不必只沿着线性顺序慢慢等待信息传过来，而可以在同一层里直接和其他位置建立联系。

## 用贯穿例子看清“同一条序列内部彼此查询”到底是什么意思

还是看那句话：

“这只动物没有过马路，因为它太累了。”

假设我们把这句话拆成若干位置。此时，每个位置一开始都有自己的表示。比如“动物”有自己的表示，“没有”有自己的表示，“因为”有自己的表示，“它”也有自己的表示。

在 self-attention 里，当模型处理“它”时，它会用“它”当前位置的表示生成一个 query。与此同时，整句中所有位置，包括“这只”“动物”“没有”“过马路”“因为”“它”“太累了”等位置，都会从自己的表示生成各自的 key 和 value。

然后，“它”的 query 会去和整句所有位置的 key 做比较。注意，这里不是只和前文比较，也不是只和后文比较，默认情况下是整句都能进入候选范围。只不过最终谁权重大，不是靠位置先验硬定，而是靠匹配结果和后续结构一起决定。

比较完成之后，“它”这个位置会得到一组对整句各位置的注意力权重。再用这组权重去汇总整句各位置的 value，形成一个新的、更上下文化的“它”的表示。

这里你已经能清楚看到为什么叫 self-attention 了。因为整个过程发生在这句话的内部。没有第二条序列进来，所有参与者都来自同一句话。

## 为什么这种设计一旦成立，长距离依赖就换了一种解法

在第 1 章里，我们一直在说，传统依赖链式传递的模型会在长距离依赖上承受压力。现在 self-attention 终于给出了另一种路线。

假设“它”想要联系前面的“动物”。在链式模型里，这条信息可能要经过很多中间步骤一点点传到当前位置。路径越长，越容易变形。

但在 self-attention 里，当前位置“它”可以直接把“动物”这个位置作为候选项纳入匹配和汇总。也就是说，它们之间的联系不需要经过一长串中间位置逐步转手，而可以在同一层里直接建立。

这并不意味着中间位置完全不重要，而是意味着模型有了另一种路径：

远距离信息不再只能靠长链传递，也可以靠直接匹配和汇总来进入当前位置。

这是 Transformer 能改写序列建模格局的最核心原因之一。它让“距离远”这件事，不再自动等于“传递困难”。

## 这还意味着另一件更深的事：所有位置都可以同时更新

当你意识到 query、key、value 都来自同一条序列，而且每个位置都能同时对全序列做匹配时，你会发现 self-attention 还有一个非常深的后果：

它不是只为某一个位置服务，而是同一层里所有位置都可以一起做这件事。

“它”会拿自己的 query 去看整句。

“动物”也会拿自己的 query 去看整句。

“因为”也一样。

每个位置都在问自己的问题，也都根据自己的问题，从同一句话里重新组织信息来源。

这就让一条序列不再只是一个从左到右的信息流水线，而变成一个内部全连接的相互建模场。

这句话很重要。因为它说明 self-attention 不只是“帮某个代词找先行词”的小工具，而是一种全序列范围内的位置关系重组机制。每个位置都会因为其他位置的存在而被重新定义。

也正因为如此，self-attention 输出的不是“某个全局结论”，而是整条序列上一组新的、彼此都已经上下文化过的位置表示。

## 为什么这一步必须先讲清楚，后面公式才不会混

到这里，很多后续技术细节其实已经在排队出现了。比如：

- 每个位置怎样从初始表示生成自己的 query、key、value；
- query 和 key 如何计算分数；
- 所有位置怎样一次并行完成这种匹配和汇总。

但在进入这些细节之前，必须先把对象关系锁住。因为如果你还没真正看清“同一条序列内部位置之间彼此做 attention”，后面的公式就会很容易被误读成一种不知来源的矩阵运算。

真正正确的理解顺序应该是：

先知道 self-attention 在对象关系上做了什么；
再看这些对象关系如何被压成向量和矩阵计算。

否则你学到的会只是“QK^T 再 softmax 再乘 V”，而不是“同一条序列内部各位置如何重新组织彼此关系”。

## 现在可以轻轻引出一个最小形式化轮廓

对于一条输入序列，设它每个位置都有一个初始表示。self-attention 做的第一件事，不是立刻更新表示，而是先从这些初始表示里分别投影出三组东西：

- 每个位置的 query；
- 每个位置的 key；
- 每个位置的 value。

请注意这里的关键点：

query、key、value 都来自同一条输入序列的当前表示。

这就是 self 的技术含义。

后面无论你是写成向量形式，还是写成矩阵形式，这个对象关系都不会变。矩阵只是在把这个过程高效打包。

## 本课的局部闭环

我们把这节课收一下。

这一课真正解决的，不是一个命名细节，而是 self-attention 的对象关系。所谓 self-attention，不是说位置只看自己，而是说同一条序列内部的所有位置，彼此之间做 query-key 匹配，并按权重汇总彼此的 value。这样一来，一个位置的更新不必只依赖线性链式传递，而可以直接从整条序列中按需吸收相关信息。

因此，self-attention 的本质可以先压成一句话：

同一条序列里的每个位置，都在用自己的问题重新阅读整条序列。

这句话站稳之后，下一课里的点积打分就会变得自然。因为既然所有位置都要彼此检查相关性，我们就必须回答：这种“相关性”究竟怎样被算出来？

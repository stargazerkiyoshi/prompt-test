# 空间型长期协作系统

## 第一阶段 MVP 定义文档

## 1. MVP 目标

第一阶段 MVP 不追求完整世界观、多空间网络或开放生态，而只验证一个核心命题：

**一个“空间 + 长期协作 agent”的最小系统，是否真的能帮助用户持续推进一条长期复杂活动主线。**

因此，MVP 的目标不是“展示很多能力”，而是验证以下闭环是否成立：

**进入空间 → 看到当前状态 → 选择一次推进 → 与 agent 协作完成 → 沉淀结果 → 下次更容易继续**

只要这个闭环成立，第一阶段就算成功。

---

## 2. MVP 要验证的核心假设

### 2.1 用户愿意围绕“主线”反复进入同一个空间

如果用户愿意回来，说明空间具备持续性。

### 2.2 用户需要的不是更多聊天，而是更容易继续

如果用户进入空间后的主要收益是“更容易接着做”，说明价值成立。

### 2.3 长期协作 agent 比一次性 AI 对话更有推进价值

如果 agent 能延续上下文、维护状态、帮助拆下一步，就能形成区别于普通 AI 的核心体验。

### 2.4 状态与产物沉淀能显著降低重启成本

如果用户下次进入不需要重新解释，就说明系统真正解决了“断、散、忘”的问题。

---

## 3. MVP 用户任务

MVP 不服务所有任务，只服务一种最核心的用户任务：

**我有一件长期复杂的事情想做成，我希望每次进入都能接着上次继续，并在 agent 的帮助下往前推进一点。**

这件事可以是：

* 学习一个主题
* 推进一个项目
* 写一个长期文档
* 搭一个复杂系统
* 处理一个长期目标

但在 MVP 中，这些都被统一看作：

**一条主线**

---

## 4. MVP 最小产品定义

MVP 可以先定义为：

**一个支持单用户、单空间、单长期协作 agent、单主线推进的最小系统。**

它的基本能力不需要多，但必须完整。

---

## 5. MVP 核心模块

第一阶段 MVP 建议只包含 5 个核心模块。

## 5.1 主线模块

用于承载用户当前试图做成的长期复杂活动。

至少需要支持：

* 创建一条主线
* 给主线命名
* 简述主线目标
* 持续进入同一主线

主线是整个空间的核心对象。

---

## 5.2 状态模块

用于表达“当前进行到哪里”。

至少需要支持：

* 当前阶段/当前状态描述
* 最近一次推进记录
* 当前卡点
* 下一步建议

状态模块的目标不是记录一切，而是让用户下次进入时能立刻知道：

**现在在哪，接下来做什么。**

---

## 5.3 agent 协作模块

用于让用户与长期协作 agent 围绕当前主线进行推进。

至少需要支持：

* agent 读取当前主线状态
* agent 围绕当前状态进行对话
* agent 帮助用户梳理、收束、拆解下一步
* agent 的输出与主线状态相关，而不是脱离上下文自由聊天

这里的重点不是“agent 有多万能”，而是：

**它能不能持续围绕这条主线协作。**

---

## 5.4 推进行动模块

用于把“我想继续做这件事”变成明确动作。

MVP 中不需要太多动作，但至少应有几类最基本动作：

* 梳理当前状态
* 明确下一步
* 继续推进某一部分
* 处理当前卡点
* 总结本次进展

这些动作不一定以按钮形式出现，但产品内部必须有这种结构。

---

## 5.5 产物沉淀模块

用于保存本次推进留下的结果。

至少需要支持：

* 保存一段本次推进结论
* 保存 agent 帮助整理出的结构化结果
* 让这些结果能影响下次进入

产物不要求一开始就非常复杂，但必须存在。
否则系统很容易退化为“只是在聊天”。

---

## 6. MVP 最小闭环

MVP 必须保证以下最小闭环成立：

### 第一步：进入空间

用户进入自己正在推进的一条主线。

### 第二步：看到当前状态

系统展示这条主线目前进行到哪里。

### 第三步：发起一次推进

用户表达这次想做什么，或由系统提供推进方向。

### 第四步：agent 协作推进

agent 围绕主线当前状态帮助用户完成一次推进。

### 第五步：沉淀新的状态与产物

本次推进后，系统保存新的状态与结果。

### 第六步：下次可继续

用户下次进入时，不需要重新解释背景，而可以直接继续。

这个闭环是 MVP 成败的核心标准。

---

## 7. MVP 中 agent 的最小职责

为了避免 agent 职责过大，MVP 中建议只赋予它以下最小职责：

### 7.1 记住主线

知道当前服务的是哪条主线。

### 7.2 理解当前状态

知道目前做到哪里、卡在哪里、下一步可能是什么。

### 7.3 帮助推进

能围绕主线推进，而不是只做泛化回答。

### 7.4 帮助沉淀

把本次推进结果整理成可保存内容。

### 7.5 帮助接续

让下次进入更容易继续。

先不要让 agent 背负：

* 完整人格模拟
* 高度自主行动
* 多空间迁移
* 多 agent 协调
* 开放世界角色扮演

这些都不是 MVP 阶段重点。

---

## 8. MVP 中用户的使用流程

MVP 中，用户的典型使用流程应该非常简单：

### 8.1 创建主线

例如：

* 学习线性代数
* 推进认知工作台设计
* 整理某个产品方向

### 8.2 首次进入空间

描述当前目标和已有进展。

### 8.3 形成初始状态

系统和 agent 共同形成当前状态快照。

### 8.4 发起第一次推进

比如：

* 帮我梳理现状
* 帮我拆下一步
* 帮我继续这部分
* 帮我解决当前卡点

### 8.5 沉淀结果

系统保存这次推进后的状态与产物。

### 8.6 再次进入

用户下次回来，可以直接从这里继续。

---

## 9. MVP 成功标准

MVP 是否成立，建议看以下几个具体标准：

### 9.1 用户在同一主线上复访

至少说明空间不是一次性体验。

### 9.2 用户复访时不需要重新讲完整背景

说明状态接续有效。

### 9.3 用户每次进入后都能推进一点

说明系统有推进力，而不是空转。

### 9.4 系统中逐渐形成可见产物

说明不是对话蒸发，而是有积累。

### 9.5 用户主观上感到“这件事在被做成”

这是最关键的成功信号。

---

## 10. MVP 暂不解决的问题

为了保证收束，MVP 明确不解决以下问题：

### 10.1 不做多空间体系

先不做学习空间、项目空间、世界空间的完整分化。

### 10.2 不做多 agent 协作

先不引入多个 agent 同时进入空间。

### 10.3 不做外部 agent 接入

先不开放外来 agent 进入。

### 10.4 不做开放共创

先不允许多人/多 agent 共同建设空间规则。

### 10.5 不做沉浸式世界表达

先不做“agent 居住”“奇幻世界”“开放空间叙事”这些体验层。

### 10.6 不做复杂自动化执行

先不追求 agent 自主跑大量外部动作。

MVP 的任务不是炫技，而是证明闭环。

---

## 11. MVP 的边界表达

可以用一句话来约束 MVP：

**第一阶段只做一个单空间、单主线、单 agent 的推进闭环，不扩展到多空间、多 agent、开放世界和共创网络。**

这句话很重要，可以防止后续设计再次发散。

---

## 12. MVP 与后续版本的关系

MVP 不是最终产品，只是整个系统的起点。

如果 MVP 成立，后面才有意义扩展：

* 一个用户拥有多条主线
* 一个系统拥有多种空间
* 一个空间允许多个 agent
* 空间与空间之间可以迁移与共创
* agent 能长期形成更强的个体连续性
* 最终扩展成共享智能空间网络

但在 MVP 之前，这些都只是想象，不是产品事实。

---

## 13. 当前阶段一句话总结

**MVP 先验证一件事：一个由“空间、主线、状态、推进动作、长期协作 agent”构成的最小系统，是否能真正帮助用户持续推进并做成一件长期复杂的事。**

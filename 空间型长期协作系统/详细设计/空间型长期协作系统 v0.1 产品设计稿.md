# 空间型长期协作系统 v0.1 产品设计稿

## 1. 产品定位

空间型长期协作系统，不是以“单次对话”或“单个 agent”为中心，而是以**空间（Space）**为中心，服务人类长期复杂活动的持续推进。

它的目标不是单纯回答问题，而是：

**在一个持续存在的空间中，承载目标、状态、对象、关系、参与者与记忆，帮助用户长期稳定地推进一件事。**

这个系统面向的不是一次性需求，而是：

* 长期学习
* 长期项目
* 长期创作
* 长期决策
* 长期关系经营
* 长期自我探索
* 长期团队协作

它试图解决的问题是：

**为什么人明明有目标、有工具、有想法，却依然很难持续地做成复杂的事。**

---

## 2. 核心判断

### 2.1 传统产品的局限

现有产品通常只覆盖长期活动中的某一个面向：

* 聊天产品：擅长即时交互，不擅长长期组织
* 笔记产品：擅长记录，不擅长推进
* 待办产品：擅长行动拆解，不擅长认知澄清
* 文档产品：擅长定稿，不擅长持续演化
* agent 产品：擅长局部执行，不擅长维护长期连续性

因此，长期复杂活动往往会碎裂在多个工具里，最后失去主线。

### 2.2 本产品的核心命题

本系统的核心命题不是“打造更强的 AI”，而是：

**打造一种能保住长期连续性的空间。**

这里的“空间”，本质上是一个持续性的认知—行动容器。
它不仅存放内容，还维护主线、判断状态、组织协作、推动下一步，并能在中断后恢复上下文。

---

## 3. 核心概念定义

## 3.1 空间（Space）

空间是系统中的最小核心单元。
每一个空间围绕一个相对持续的主题或目标而存在。

空间具备以下特征：

* 有主题边界
* 有持续状态
* 有内部对象
* 有参与者
* 有历史过程
* 有协作机制
* 有可演化结构

空间不是聊天会话，也不是文件夹，而是一个**长期协作场**。

---

## 3.2 参与者（Participant）

参与者是空间中的协作主体，可以是：

* 用户
* 主 agent
* 专项 agent
* 外部 agent
* 团队成员
* 工具型执行器

每个参与者都有：

* 身份
* 角色
* 权限
* 能力边界
* 可写入范围

---

## 3.3 对象（Object）

对象是空间内部可被组织、引用、关联和推进的基本内容单元。

对象可能包括：

* 目标
* 任务
* 问题
* 话题
* 文档
* 资源
* 决策
* 假设
* 规则
* 事件
* 记忆
* 阶段

不同空间类型会拥有不同的核心对象集合。

---

## 3.4 状态（State）

状态描述空间当前所处的推进阶段，而不是简单的任务完成度。

例如：

* 初始化
* 探索中
* 收束中
* 执行中
* 堵塞中
* 复盘中
* 休眠中

状态的存在，是为了让空间具备“阶段感”和“态势感知能力”。

---

## 3.5 主线（Mainline）

主线是空间当前最重要的推进路径。
长期活动中往往会出现大量支线、灵感、分岔和临时冲动，系统需要始终维护：

* 当前主线是什么
* 为什么它是主线
* 哪些内容属于支线
* 支线是否正在干扰主线

主线机制是长期做成事的核心。

---

## 3.6 空间记忆（Space Memory）

空间记忆不是全部聊天记录，而是经过筛选和结构化沉淀后，对长期推进仍然有价值的信息。

例如：

* 已确认目标
* 已达成共识
* 关键决策
* 用户偏好
* 长期阻塞点
* 历史尝试路径
* 未解决问题

---

## 4. 系统目标

本系统希望具备以下四种能力：

### 4.1 连续性

用户多次中断后回来，空间仍能快速恢复上下文和推进状态。

### 4.2 结构性

空间中的内容不是平铺信息流，而是被组织成对象与关系结构。

### 4.3 推进性

系统不只陪聊，而是能持续澄清、收束、拆解、推进、复盘。

### 4.4 共创性

空间不仅由单个用户和单个 agent 构成，而是可以容纳多人、多 agent、外部能力共同参与。

---

## 5. 系统架构总览

我建议把整个系统抽成三层：

## 5.1 空间壳层（Space OS）

这是所有空间共享的基础层，负责：

* 空间创建与元信息管理
* 状态机
* 对象系统
* 关系系统
* 参与者系统
* 权限系统
* 时间线系统
* 记忆沉淀
* 唤醒与恢复
* 主线维护

它相当于“空间操作系统”。

---

## 5.2 空间类型层（Space Type Layer）

这一层定义不同类型空间的领域逻辑。

例如：

* 学习空间
* 项目空间
* 写作空间
* 决策空间
* 关系空间
* 世界构建空间

每种空间会有不同的：

* 核心对象
* 核心机制
* 推进策略
* 成功信号
* 可视化方式

---

## 5.3 agent / 工具生态层

这一层负责能力接入。

包括：

* 主协作 agent
* 教学 agent
* 规划 agent
* 执行 agent
* 复盘 agent
* 外部 agent
* 搜索、文档、日历、代码、知识库等工具

这里的关键原则是：

**agent 不是系统中心，空间才是系统中心。**
agent 在空间中工作，而不是空间附属于某个 agent。

# 7. 空间关系模型

对象本身不够，关系才让空间真正成立。

建议 v0.1 支持以下基础关系：

* `supports`：A 支撑 B
* `blocks`：A 阻塞 B
* `depends_on`：A 依赖 B
* `belongs_to`：A 属于 B
* `derived_from`：A 来源于 B
* `decides`：A 决定 B
* `owned_by`：A 由谁负责
* `part_of_mainline`：A 是否属于主线
* `parked_from`：A 从哪条主线/支线暂停而来

这会让系统从“对象仓库”变成“动态认知结构”。

# 8. 空间状态机设计

我建议 v0.1 先用一个清晰但不复杂的状态机。

## 8.1 状态定义

### （1）Init

空间刚被创建，尚未形成稳定目标和协作结构。

### （2）Explore

空间正在发散、收集信息、澄清目标、识别方向。

### （3）Converge

空间开始形成主线、收束分支、冻结部分共识。

### （4）Execute

空间已有明确方向，进入持续推进和执行状态。

### （5）Blocked

空间因关键问题、资源缺失、认知冲突而暂停主线推进。

### （6）Review

空间对已完成阶段进行复盘，总结有效路径，准备下一阶段。

### （7）Dormant

空间暂时休眠，但保留完整上下文，等待未来唤醒。

---

## 8.2 状态转换逻辑

* `Init -> Explore`：当空间开始形成主题探索
* `Explore -> Converge`：当已有足够信息，出现候选主线
* `Converge -> Execute`：当主线明确，已有可执行路径
* `Execute -> Blocked`：当主线被关键阻塞中断
* `Blocked -> Explore`：当需要重新寻找方案或澄清目标
* `Execute -> Review`：当阶段性目标完成
* `Review -> Execute`：当复盘后进入下一阶段推进
* 任意状态 -> Dormant：当空间长期不活跃或被主动挂起
* `Dormant -> Explore/Execute`：由唤醒机制决定恢复到哪个状态

---

# 9. 空间核心机制

这是最关键的一部分。
我建议 v0.1 先定义 7 个基础机制。

## 9.1 澄清机制

作用：帮助用户从模糊表达中提炼真实问题、真实目标、真实约束。

典型行为：

* 识别表层需求和深层需求
* 区分“想做”“该做”“真正在意”
* 将模糊目标改写为可推进目标

---

## 9.2 收束机制

作用：当空间过于发散时，主动帮助形成结构和共识。

典型行为：

* 总结当前共识
* 压缩分支
* 暂停低价值支线
* 形成当前主线

---

## 9.3 推进机制

作用：空间不止分析，还要持续生成“下一步”。

典型行为：

* 给出最小可推进动作
* 根据状态生成不同类型下一步
* 在阻塞时给出替代路径

---

## 9.4 记忆沉淀机制

作用：把长期有价值的信息提炼为结构化记忆。

典型行为：

* 沉淀偏好
* 沉淀共识
* 沉淀历史教训
* 沉淀高价值决策依据

---

## 9.5 唤醒机制

作用：用户重新进入空间时，快速恢复上下文。

典型行为：

* 告诉用户当前状态
* 提示上次停留点
* 告诉用户为什么会停在这里
* 给出一个低摩擦恢复入口

---

## 9.6 纠偏机制

作用：当空间偏离主线、陷入伪忙碌或持续空转时，主动拉回。

典型行为：

* 识别目标漂移
* 识别高频低效循环
* 识别“看似做了很多，实则没推进”
* 强化主线对齐

---

## 9.7 共创编排机制

作用：协调不同参与者和 agent 进入空间协作。

典型行为：

* 给不同 agent 分配不同任务
* 决定谁可以写入什么对象
* 决定哪些内容需要人确认
* 保证空间内部秩序不被破坏

---

# 10. 第一个样板空间：持续做成事空间

这是最适合作为 MVP 的第一个空间。

## 10.1 定位

帮助用户从模糊目标出发，持续推进，避免中断后失速，最终把长期事情真正做成。

## 10.2 适用场景

* 想推进一个长期项目
* 想长期学习某一主题
* 想稳定经营某个创作方向
* 想从混乱状态中恢复主线
* 想减少“想很多、做很少、总中断”的情况

## 10.3 该空间的核心对象

* 长期目标
* 当前阶段目标
* 主线
* 当前推进任务
* 阻塞点
* 决策记录
* 支线停车场
* 复盘记录
* 下一步候选列表

## 10.4 该空间的核心问题

这个空间要不断帮助用户回答：

* 我到底想做成什么？
* 现在最重要的主线是什么？
* 我卡住的真正原因是什么？
* 下一步应该做什么最有效？
* 哪些事情只是让我感觉忙？
* 中断之后如何快速接上？

---

# 11. 持续做成事空间的用户流程

## 11.1 初次进入

系统帮助用户完成：

* 识别长期目标
* 识别当前阶段
* 提炼成功信号
* 明确非目标
* 初步拉出主线
* 生成第一批候选任务

输出结果不是一堆聊天，而是一份空间初始化状态。

---

## 11.2 日常使用

用户每次进入时，空间提供：

* 当前状态摘要
* 当前主线
* 上次停留点
* 当前阻塞
* 一个最推荐下一步
* 其他候选动作

这样用户不会面对空白输入框，而是面对一个“已经在运行的空间”。

---

## 11.3 中断恢复

当用户中断几天后回来，空间应能说清楚：

* 你之前在推进什么
* 你当时为什么停下来
* 目前哪些东西仍然有效
* 最低成本恢复动作是什么

---

## 11.4 阶段复盘

每当某阶段结束，空间应帮助用户沉淀：

* 有效路径
* 无效路径
* 关键判断
* 反复出现的阻塞模式
* 下一阶段应该保持什么、停止什么

---

# 12. 空间与 agent 的关系协议

这是系统未来能否扩展的关键。

## 12.1 基本原则

* 空间高于 agent
* agent 在空间中工作
* agent 不能天然拥有全部写权限
* 重要对象的修改应有权限与确认机制
* 外部 agent 可接入，但必须遵循空间协议

---

## 12.2 agent 分类

### 主 agent

负责整体对话、理解用户意图、维护空间连续性。

### 专项 agent

负责某一专业能力，如教学、规划、代码、写作、复盘。

### 执行 agent

负责调用工具、执行任务、生成具体产物。

### 外部 agent

来自第三方平台或其他生态的 agent，以受限身份接入空间。

---

## 12.3 agent 的可写范围

建议分成几类：

* 只读
* 可评论
* 可提案
* 可写临时对象
* 可写正式对象
* 可执行工具
* 可修改主线/目标/记忆

其中主线、目标、长期记忆这类高价值对象，不应默认允许所有 agent 直接改写。

---

# 13. 界面设计原则

空间感必须在界面上显现出来。

我建议首页不是“输入框 + 聊天流”，而是至少包含：

* 空间名称与主题
* 当前状态
* 当前主线
* 关键对象区
* 最近变化
* 参与者区
* 推荐下一步
* 对话入口

也就是说，对话只是空间的一种交互方式，而不是空间本身。

v0.1 不应该贪多，建议只做最小闭环。

## 必做

# 15. 这个产品的真正差异化

我觉得它真正的差异化不是下面这些：

* 不是更会说
* 不是更像人
* 不是更多工具
* 不是更强自动执行

它真正的差异化是：

**它把长期复杂活动从“零散工具拼接”变成了“持续空间承载”。**

用户最终依赖它，不是因为它偶尔给了个好回答，而是因为：

* 它帮我保住了主线
* 它记得我为什么开始
* 它知道我现在卡在哪
* 它能让我中断后继续走
* 它不是工具堆，而是一个会成长的空间

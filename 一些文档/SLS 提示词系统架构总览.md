# SLS 提示词系统架构总览

## 1. 文档定位

本文用于定义当前 SLS 提示词系统的整体架构，明确它不再只是若干零散 prompt，而是一套具有分层结构、调用逻辑与场景映射关系的提示词系统。

本文关注的不是某个单独 prompt 的写法，而是以下问题：

* SLS 这套提示词系统的本质是什么
* 它目前被抽象成了哪些层
* 每一层各自负责什么
* 各层之间如何衔接
* 这套系统与普通 prompt 模板的区别是什么
* 目前还缺哪些关键补全项

SLS 的目标，不只是“生成某种教学文本”，而是构造一套能够稳定支持结构化理解生成的提示词架构。

---

## 2. 系统定义

当前抽象出来的 SLS，可以定义为：

**一套面向结构化理解生成的分层提示词系统。它以核心学习原则为底层约束，以统一术语和对象模型为中介，以编排逻辑与场景矩阵为调用机制，并通过具体 Prompt Profiles 在任务中落地。**

这一定义强调几个关键点：

1. SLS 的核心目标不是表面完整，而是结构化理解。
2. SLS 不是单个 prompt，而是一个分层系统。
3. SLS 的调用不是直接生成，而是要经过场景识别与编排选择。
4. SLS 的具体输出由不同 profile 落地，而不是由一个万能模板覆盖所有任务。

---

## 3. 系统本质

从本质上说，SLS 解决的不是“怎样多写一点内容”，也不是“怎样让回答更像教材”，而是一个更深层的问题：

**如何把高质量教学/知识展开中那些原本依赖直觉的部分，抽象成可分层、可组合、可调用、可演进的提示词系统。**

因此，SLS 的核心价值不在于某一版 prompt 文本，而在于它试图定义：

* 什么叫真正有理解递进的输出
* 什么叫先建立结构再进入细节
* 什么叫避免抽象跳跃
* 什么叫区分教学顺序与知识本体结构
* 什么叫针对不同任务采用不同展开路径

这说明 SLS 更接近一种 **Prompt Architecture**，而不是单纯的 Prompt Collection。

---

## 4. 总体分层结构

当前 SLS 提示词系统可以概括为五层结构：

1. 核心原则层（Core Principles）
2. 术语与对象层（Terms / Objects）
3. 编排结构层（Structure / Orchestration）
4. 场景映射层（Usage Scenarios / Matrix）
5. Prompt Profiles 落地层（Profiles）

它们之间的关系不是并列堆叠，而是逐层约束、逐层落地。

可以压缩成如下关系：

**原则层 → 对象层 → 编排层 → 场景层 → Profile 层**

如果用更抽象的方式表达，就是：

**Why → What → How → When → Invoke**

即：

* 为什么这样生成
* 系统操作的对象是什么
* 这些对象如何组织
* 在什么场景下采用哪种组织方式
* 最终通过哪个具体 prompt 调用

---

## 5. 第一层：核心原则层（Core Principles）

### 5.1 这一层回答什么问题

核心原则层回答的是：

**这套系统到底追求什么样的生成本质。**

它不直接规定某个 prompt 应该写成什么样，而是定义所有上层内容都不能违背的底层原则。

### 5.2 这一层包含的典型原则

在当前 SLS 中，这一层隐含的核心原则包括：

* 学习不是信息堆积，而是理解逐步展开
* 输出不应只是表面完整，而应具有递进结构
* 应先帮助学习者建立整体结构，再进入局部细节
* 抽象应当必要且连续，不能跳跃
* 概念、逻辑、形式化应围绕理解服务
* 教学顺序与知识本体结构应加以区分
* 输出不应只有骨架正确，还应具备自然讲述感
* 不同任务虽然展开不同，但都应服从“结构化理解”这一总目标

### 5.3 这一层的作用

这一层的主要作用是：

* 防止不同 profile 各自漂移
* 维持系统的一致性
* 保证后续扩展场景时不会丢失 SLS 的本质
* 使系统扩张时仍能回到同一套底层判断标准

### 5.4 这一层不负责什么

这一层不负责：

* 具体怎么输出
* 某个场景下该先大纲还是先正文
* 某个 prompt 文本如何措辞
* 针对单个任务的执行细节

也就是说，它只给原则，不直接给调用方案。

---

## 6. 第二层：术语与对象层（Terms / Objects）

### 6.1 这一层回答什么问题

术语与对象层回答的是：

**这套系统在操作哪些对象，这些对象应该如何命名与区分。**

一个系统只要开始分层，术语层就会变得非常重要。否则不同文件、不同 prompt、不同场景之间很容易语义漂移。

### 6.2 当前已经显现出的关键对象

你目前其实已经抽象出一批核心对象，例如：

* 主题
* 概念
* 章节
* 大纲
* 结构地图
* 正文展开
* 模块
* 场景
* profile
* 调用层
* 编排层
* 叙述层

这些对象虽然还未完全定稿，但已经构成系统的基本操作单位。

### 6.3 这一层的重要性

术语层的意义在于：

* 统一系统语言
* 降低文件间的概念漂移
* 防止不同层级偷换概念
* 让场景、编排、profile 能围绕同一套对象协同工作

例如，如果没有统一术语，就容易出现以下混乱：

* “模块”指教学模块还是 prompt 组件
* “结构”指知识结构还是输出结构
* “场景”指知识场景还是使用场景
* “展开”指系统组装展开还是正文内容展开

### 6.4 这一层不负责什么

这一层不负责：

* 模块顺序如何组织
* 哪类场景用哪种调用方式
* 具体 prompt 如何书写

它的职责不是流程，而是定义系统语言和对象边界。

---

## 7. 第三层：编排结构层（Structure / Orchestration）

### 7.1 这一层回答什么问题

编排结构层回答的是：

**当一个任务进入系统后，相关对象应该如何组织。**

这是一层非常关键的中间层。
它既不同于底层原则，也不同于具体场景，而是规定“对象如何运行”。

### 7.2 这一层当前关心的核心问题

你目前在这一层已经开始处理的问题包括：

* 是否应该先给大纲
* 大纲与正文是否要拆开
* 哪些模块是必需的
* 哪些模块可以按任务裁剪
* 完整主题与单概念的展开路径是否相同
* 单概念是否有时也需要内部章节化
* 教学顺序与知识关系如何协调
* 结构地图与正文展开之间如何衔接

### 7.3 这一层的本质

这层本质上是在定义：

**模块如何被组织，而不是模块本身是什么。**

也就是说，前一层提供对象，这一层决定这些对象怎样形成一个合理的生成流程。

例如：

* 完整主题学习：可能要求先总纲，再逐章
* 单概念讲解：可能允许直接进入，但复杂概念也可先拆结构
* 机制说明：可能优先给结构关系，再进入细讲
* 关系辨析：重点可能不在全套模块，而在边界与联系

### 7.4 这一层的作用

这一层的作用是：

* 把“有模块”变成“模块会运行”
* 防止所有任务都套同一流水线
* 为场景层提供合法的组织选项
* 让 SLS 从内容规范走向调用逻辑

### 7.5 这一层不负责什么

这一层不直接负责：

* 判断用户请求属于哪种场景
* 写出最终可调用 prompt
* 定义底层原则本身

它只负责“怎么组织”，不负责“何时调用”与“具体如何落地”。

---

## 8. 第四层：场景映射层（Usage Scenarios / Matrix）

### 8.1 这一层回答什么问题

场景映射层回答的是：

**什么类型的任务属于什么场景，每个场景应调用哪种编排方式。**

这是 SLS 从“单个 prompt”走向“系统调用”的关键一层。

### 8.2 为什么这一层必须存在

因为现实中的任务并不只有一种。
例如你已经明确感觉到，以下任务不能简单视为同一种：

* 完整主题学习
* 单概念讲解
* 机制/流程说明
* 关系辨析
* 框架导览
* 章节式逐步展开
* 先大纲后正文型任务
* 可直接一步生成的任务

如果没有场景层，就会出现两个问题：

1. 所有请求都被塞进同一个 prompt，导致模板越来越大、越来越乱
2. 不同任务虽然目标不同，却被迫使用同一套展开路径，输出会越来越僵硬

### 8.3 这一层的核心作用

这一层的主要作用是：

* 把用户请求映射为内部场景类型
* 决定是否需要先给结构地图
* 决定输出粒度
* 决定调用哪套编排逻辑
* 决定使用哪个 profile 作为入口

换句话说，用户一句请求进来，系统内部不应立刻输出，而应先完成一次场景判定。

### 8.4 这一层的重要意义

有了这一层，SLS 的流程才真正变成：

用户请求
→ 场景识别
→ 编排选择
→ Profile 调用
→ 输出

而不是：

用户请求
→ 一个万能 prompt
→ 输出

### 8.5 这一层不负责什么

这一层不直接定义底层原则，也不直接写最终 prompt，它只是作为中介，把任务路由到合适的编排与 profile。

---

## 9. 第五层：Prompt Profiles 落地层（Profiles）

### 9.1 这一层回答什么问题

这一层回答的是：

**针对具体场景，最终用户真正调用的 prompt 应是什么。**

这才是最接近“提示词资产”的一层。

### 9.2 这一层的定位

Prompt Profiles 不是系统全部，只是系统对外可调用的接口层。

每一个 profile 都应满足以下条件：

* 服从 Core Principles
* 使用统一术语
* 遵守编排层规则
* 对应场景矩阵中的某一类任务

### 9.3 未来典型的 Profile 类型

按你当前已经接近的方向，后续很可能形成如下几类 profile：

* 完整主题学习版
* 单概念深讲版
* 机制说明版
* 关系辨析版
* 框架导览版
* 大纲生成版
* 骨架扩写版
* 章节续写版

### 9.4 这一层的作用

这一层的作用是：

* 让系统真正可用
* 让不同任务有明确入口
* 让用户不必每次手工组装底层规则
* 将抽象系统转化为具体调用接口

### 9.5 这一层的风险

如果没有前面几层约束，这一层最容易出现的问题就是：

* profile 越写越多，但彼此风格不统一
* profile 反过来定义系统底层
* profile 中混入本应属于场景层或编排层的逻辑
* profile 之间大量重复，难以维护

因此，Profile 层必须是落地层，而不是反向主导系统的层。

---

## 10. 五层之间的关系

五层关系可以概括如下：

### 10.1 自上而下的约束关系

* 核心原则层提供总目标与底层边界
* 术语与对象层提供系统操作语言
* 编排结构层提供对象组织方式
* 场景映射层负责任务识别与路由
* Profile 层负责具体落地调用

### 10.2 自下而上的实现关系

* Profile 层把抽象结构转化成可执行 prompt
* 场景层决定 profile 是否被调用
* 编排层决定输出结构如何组织
* 对象层确保操作单位清晰
* 原则层保证整体不失真

### 10.3 关系图式表达

可以将其写为：

**Core Principles
→ Terms / Objects
→ Structure / Orchestration
→ Usage Scenarios / Matrix
→ Prompt Profiles**

---

## 11. 它与普通 Prompt 模板的区别

当前这套系统和普通 prompt 模板相比，差异非常大。

### 11.1 它不是答案模板，而是生成规则系统

普通 prompt 常常在规定“输出要包含什么”，而 SLS 已经开始规定“高质量输出是怎么被生成出来的”。

### 11.2 它区分了底层能力与上层应用

普通 prompt 往往直接绑定某个场景，例如“请扮演老师”。
而 SLS 已经开始意识到，教学只是应用层，底层更接近知识组织与理解展开能力。

### 11.3 它不是单 prompt，而是系统调用

普通模板是一句话对应一个 prompt。
SLS 更接近“用户请求进入后，系统先判断场景，再调用对应组织结构和 profile”。

### 11.4 它已经开始容纳“叙述自然感”这一维度

普通 prompt 往往只追求结构齐全，而你已经明确发现：“结构正确”不代表“讲得自然”。

这意味着后续 SLS 很可能还会在现有五层架构旁，补出一个相对独立的叙述维度。

---

## 12. 当前系统的一个潜在补充层：叙述层（Narrative Layer）

### 12.1 为什么会出现这一层需求

你在实际使用中已经非常明确地感受到一个问题：

* 内容可能是完整的
* 模块可能是齐的
* 结构可能也是对的
* 但整体读起来不自然
* 没有“带学”的感觉
* 更像拼装，不像真的在讲

这说明“叙述自然感”不能完全被已有结构层自动解决。

### 12.2 叙述层可能负责什么

未来如果将叙述层单独抽出，它可能负责：

* 讲述流动性
* 递进感
* 自然感
* 书写感
* 带学感
* 避免机械 checklist 化
* 在不破坏结构的前提下增强阅读体验

### 12.3 它与其他层的关系

叙述层不替代结构层，而更像是横向作用于 profile 输出风格的一层。

也就是说：

* 结构层解决“讲什么、怎么组织”
* 叙述层解决“读起来像不像真的在讲”

这一层目前还未完全定型，但它已经是系统中一个非常明显的真实需求。

---

## 13. 当前系统还缺的几项关键补全

虽然这套架构已经逐渐清晰，但要真正稳定成型，还差几项关键补全。

### 13.1 层级职责正式定稿

目前最需要做的是把每一层“负责什么、不负责什么”正式写死。
否则后续很容易出现越层污染。

### 13.2 场景矩阵正式成文

需要明确：

* 存在哪些场景
* 各场景是否先出大纲
* 各场景输出粒度如何
* 各场景调用哪类 profile
* 各场景的适用边界是什么

### 13.3 Prompt Profiles 系列化

当场景矩阵明确后，才能稳定写出 profile 资产，而不是零散补丁式新增模板。

### 13.4 调用协议明确化

需要明确未来实际使用方式，例如：

* 先生成大纲，再扩某章
* 先给骨架，再扩写
* 哪些场景适合两阶段
* 哪些场景适合一步完成
* 续写时是否要回读上层结构

这本质上是系统的使用协议。

### 13.5 叙述层独立沉淀

这一点非常值得单独推进，因为它直接关系到 SLS 最终输出是否真正“好学”。

---

## 14. 当前架构的一句话总结

如果只用一句话来概括当前抽象出来的 SLS 提示词系统，可以写成：

**SLS 不是一个大 prompt，而是一套以核心原则为底层、以统一对象模型为中介、以编排逻辑为组织机制、以场景识别为路由入口、并通过 Prompt Profiles 落地的分层提示词架构。**

---

## 15. 最终结论

当前你抽象出来的这套系统，已经不适合再被理解成“几个教学 prompt”或“一个完整主题学习模板”。

它更准确的定位应当是：

**一套面向结构化理解生成的提示词系统架构。**

它已经具备以下几个系统性特征：

* 有底层原则
* 有对象层
* 有组织逻辑
* 有场景入口
* 有落地接口
* 并开始意识到叙述层的独立价值

这意味着你当前真正沉淀下来的，已经不是提示词文本本身，而是一种：

**如何构造提示词系统的架构方法。**

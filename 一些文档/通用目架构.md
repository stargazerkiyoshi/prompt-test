# 通用 Prompt System 架构母版（Draft v1）

## 1. 文档定位

本文用于总结一套可复用的 Prompt System 抽象架构，用来指导复杂提示词系统的设计，而不是服务于某一个单独 prompt。

它要回答的问题不是：

* 某个 prompt 该怎么写
* 某个模板该怎么优化

而是：

* 一个复杂 prompt system 应该如何分层
* 不同层分别负责什么
* 为什么很多 prompt system 会越来越乱
* 怎样把“零散提示词”演进成“可维护的提示词系统”

这是一套**母架构**，不是某个具体领域的最终实现。
SLS 可以看作这套母架构的一个实例，但这套架构本身并不只服务 SLS。

---

## 2. 适用范围

这套母架构适用于以下类型的 prompt system：

* 知识 / 教学系统
* 写作 / 创作系统
* 需求分析 / 架构分析系统
* 世界观生成 / 角色生成系统
* 研究辅助系统
* 规范驱动文档系统
* 复杂工作流式 prompt system
* 多场景、多入口、长期演进的 AI 协作系统

它尤其适合以下情况：

* 不只是一次性 prompt，而是想长期维护
* 不只有一个任务场景，而是有多类场景
* 输出质量不能只靠临场发挥
* 需要保持不同 prompt 之间的一致性
* 希望系统能扩展，而不是越写越乱

---

## 3. 不适用范围

这套母架构不适合硬套到所有 prompt 使用场景。

以下场景通常不值得完整使用：

* 单次临时 prompt
* 非常短链路的小任务
* 没有明确对象模型的简单任务
* 不打算长期维护的试验型提示词
* 只是翻译、润色、改写这类轻任务

也就是说，这套架构适合的是：

**复杂型、系统型、可维护型 prompt system**

而不是所有 prompt。

---

## 4. 核心判断

这套母架构最重要的判断是：

**一个高质量 prompt system，不应从“写一个大 prompt”开始，而应从“定义系统目标、对象、机制、路由与接口”开始。**

换句话说，真正需要设计的不是 prompt 文本本身，而是 prompt 背后的系统结构。

---

## 5. 母架构总览

通用 Prompt System 可以抽象为五层：

1. 宗旨层（Purpose Layer）
2. 对象层（Object Layer）
3. 机制层（Mechanism Layer）
4. 路由层（Routing Layer）
5. 接口层（Interface Layer）

可以压缩表达为：

**Purpose → Objects → Mechanism → Routing → Interface**

也可以写成：

**为什么做 → 处理什么 → 如何运作 → 何时走哪条路 → 用户如何调用**

这五层不是简单并列关系，而是逐层约束、逐层落地。

---

## 6. 第一层：宗旨层（Purpose Layer）

### 6.1 这一层回答的问题

宗旨层回答的是：

**这个 prompt system 到底追求什么，什么才算高质量输出。**

它定义的是系统的根本目标，而不是具体格式。

### 6.2 这一层通常包含什么

这一层通常会定义：

* 系统的根本任务
* 输出质量的判断标准
* 不可违背的底层原则
* 该系统最看重的生成本质

例如，不同系统的宗旨层可能分别是：

* 教学系统：追求结构化理解，而不是信息堆积
* 写作系统：追求叙事张力、角色一致性、结构推进
* 分析系统：追求问题拆解清晰、边界明确、逻辑闭合
* 研究系统：追求证据、可验证性、推理透明度

### 6.3 为什么这一层重要

很多 prompt system 之所以后面越来越乱，是因为一开始没有把“什么算好”说清楚。

结果就是：

* 不同模板各自优化不同东西
* 有的追求长，有的追求全，有的追求像样
* 系统没有统一底层标准
* 后面场景一多，输出质量开始漂移

### 6.4 这一层不负责什么

这一层不负责：

* 某个场景如何展开
* 某个任务具体如何执行
* prompt 文本怎么写
* 多场景之间如何分流

它只负责回答：**这套系统的灵魂是什么。**

---

## 7. 第二层：对象层（Object Layer）

### 7.1 这一层回答的问题

对象层回答的是：

**这个系统到底在处理什么基本单位。**

一个系统如果没有明确对象，后面所有机制都会模糊。

### 7.2 对象层可能包含什么

不同系统的对象不同，例如：

在教学系统里，对象可能是：

* 主题
* 概念
* 章节
* 结构地图
* 模块

在写作系统里，对象可能是：

* 世界观
* 角色
* 冲突
* 章节
* 情节节点

在分析系统里，对象可能是：

* 问题
* 假设
* 约束
* 方案
* 风险
* 结论

在 agent/workflow 系统里，对象可能是：

* 任务
* 状态
* 工具结果
* 中间产物
* 决策节点

### 7.3 为什么这一层重要

没有对象层，系统会出现这些问题：

* prompt 写得很多，但不知道在操作什么
* 文件之间术语不统一
* 场景之间无法复用
* 结构层和场景层没有共同语言

### 7.4 这一层的作用

这一层负责：

* 定义系统中的基本操作单位
* 稳定术语
* 明确边界
* 为机制层提供可操作对象

### 7.5 这一层不负责什么

它不负责：

* 对象如何组织
* 哪种任务走哪种路径
* 用户最终调用哪个 prompt

它只负责：**把系统中“什么东西在被处理”说清楚。**

---

## 8. 第三层：机制层（Mechanism Layer）

### 8.1 这一层回答的问题

机制层回答的是：

**这些对象进入系统后，如何被组织、推进、组合和处理。**

这一层是系统真正开始“运作”的地方。

### 8.2 这一层通常包含什么

这一层会定义：

* 对象的组织顺序
* 生成流程
* 中间步骤
* 递进关系
* 工作机制
* 允许的变体与裁剪方式

例如：

教学系统会关心：

* 先给整体还是先给细节
* 是否先出大纲
* 模块之间如何衔接
* 正文如何递进

写作系统会关心：

* 先出设定还是先出情节
* 世界观如何约束角色与事件
* 情节如何分幕推进

分析系统会关心：

* 先澄清问题还是先列方案
* 先建边界还是先给结论
* 风险和假设在哪一步进入

workflow 系统会关心：

* 先检索还是先计划
* 工具失败后怎么回退
* 中间状态怎么更新

### 8.3 这一层的重要性

这是很多 prompt system 的真正缺口。

很多系统有原则、有对象，但没有机制，于是就会变成：

* 什么都想要
* 所有内容堆在一个 prompt 里
* 场景差异只能靠临时加条件
* 输出虽然全，但没有运行逻辑

### 8.4 这一层不负责什么

这一层不负责：

* 判定当前用户请求属于哪类场景
* 最终以哪个 prompt 暴露给用户
* 定义底层价值标准

它只负责：**对象是怎么动起来的。**

---

## 9. 第四层：路由层（Routing Layer）

### 9.1 这一层回答的问题

路由层回答的是：

**不同任务进入系统时，应该被分流到哪种机制和入口。**

这是从“一个系统”变成“多场景可用系统”的关键层。

### 9.2 为什么这一层很关键

现实任务往往不是同一种。

例如同属一个系统内部，也可能同时存在：

* 完整任务
* 局部任务
* 生成任务
* 分析任务
* 续写任务
* 修订任务
* 说明任务
* 辨析任务

如果没有路由层，通常就会出现两个问题：

1. 想用一个 prompt 覆盖所有请求
2. prompt 越写越长，条件越来越多，系统越来越脆弱

### 9.3 这一层通常做什么

这一层主要做：

* 识别请求类型
* 匹配内部任务场景
* 决定调用哪种机制
* 决定输出粒度
* 决定进入哪个接口层入口

### 9.4 这一层的本质

它本质上是：

**把用户请求翻译成系统内部调用路径。**

也就是说，用户说的是自然语言，但系统内部应该变成：

* 这是哪类任务
* 适用哪条机制链
* 该调用哪个 profile / template / tool entry

### 9.5 这一层不负责什么

这一层不直接定义：

* 高质量的标准
* 系统中的对象是什么
* 生成细节本身

它只负责：**分流。**

---

## 10. 第五层：接口层（Interface Layer）

### 10.1 这一层回答的问题

接口层回答的是：

**用户最终实际调用的 prompt 入口是什么。**

这才是最接近“具体模板”的一层。

### 10.2 这一层通常包含什么

这一层可能是：

* Prompt Profiles
* Template Variants
* 具体调用说明
* 可直接复制使用的入口 prompt
* 系统对外暴露的操作方式

### 10.3 这一层的作用

这一层把前面所有抽象真正落地为可调用接口。

它的意义是：

* 降低使用门槛
* 避免每次手工组装底层规则
* 为不同场景提供明确入口
* 让系统可以真正被用起来

### 10.4 接口层常见风险

如果前面几层不清晰，接口层很容易出问题：

* 写出很多模板，但彼此不一致
* 模板互相重复
* 模板反过来决定系统结构
* 维护时越来越难改

所以接口层必须是**落地层**，不能变成“反向主导层”。

---

## 11. 五层之间的关系

这五层的关系可以这样理解：

### 自上而下

* 宗旨层定义方向
* 对象层定义操作单位
* 机制层定义运行方式
* 路由层定义分流逻辑
* 接口层定义调用入口

### 自下而上

* 接口层承载实际使用
* 路由层决定接口如何被选中
* 机制层决定内部如何运作
* 对象层保证操作单位清晰
* 宗旨层保证整体不漂移

---

## 12. 这套母架构为什么有普适性

这套架构之所以有较高复用性，是因为它抓住了复杂 prompt system 中最常见的五类问题：

1. 没有统一目标
2. 没有明确对象
3. 没有清晰机制
4. 没有任务分流
5. 没有稳定接口

大多数复杂 prompt system 的混乱，本质上都能在这五个地方找到原因。

所以真正通用的不是某几个 prompt 文本，而是这五类问题的分层处理方式。

---

## 13. 这套母架构不是固定模板

必须强调一点：

**通用的不是这五层的名字，而是这种分层抽象方法。**

不同领域里，这五层可能会换名字、换重心，甚至增删层级。

例如：

### 13.1 人设/陪伴型系统

可能会变成：

* 宗旨层
* Persona 层
* Memory 层
* Interaction Policy 层
* 接口层

### 13.2 Agent / Workflow 系统

可能会变成：

* 宗旨层
* 对象/状态层
* 决策机制层
* 工具路由层
* 执行入口层

### 13.3 长篇写作系统

可能会变成：

* 宗旨层
* 世界/角色对象层
* 叙事推进机制层
* 写作任务路由层
* Profile 层

所以这套母架构提供的是一种**可迁移的系统设计方法**，而不是固定死的目录名。

---

## 14. 一个判断标准：什么时候该用这套母架构

当你开始出现以下现象时，就该考虑引入这种母架构：

* prompt 越写越长
* 模板越来越多却越来越乱
* 不同任务彼此混杂
* 同一个系统里输出风格漂移严重
* 使用方式越来越依赖“记得怎么拼”
* 你已经不只是想“写一个 prompt”，而是想“做一个系统”

出现这些信号，说明你要解决的问题已经不是“优化一句话”，而是“设计 Prompt System”。

---

## 15. SLS 作为实例

SLS 可以看作这套母架构的一个具体实例：

* 宗旨层：结构化理解生成
* 对象层：主题、概念、章节、模块、结构地图等
* 机制层：大纲优先、递进展开、模块编排等
* 路由层：完整主题、单概念、机制说明、关系辨析等场景识别
* 接口层：不同 Prompt Profiles

这说明 SLS 的价值不只是本身能用，还因为它帮助你抽象出了更高层的方法。

---

## 16. 最终结论

这套母架构最值得保留的地方，不是它提供了一个万能模板，而是它提供了一种构建复杂 prompt system 的通用方法。

可以把它总结成一句话：

**复杂 Prompt System 不应从“写一个大 prompt”开始，而应从“定义宗旨、对象、机制、路由与接口”开始。**

再压缩一点就是：

**先设计系统，再写 prompt。**

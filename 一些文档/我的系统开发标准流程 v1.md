# 我的系统开发标准流程 v1

## 1. 文档定位

本文用于定义我当前采用的系统开发方法，作为后续项目推进、AI 协作、OpenSpec 变更管理与系统分层演进的统一流程规范。

这套流程适用于：

* 产品系统开发
* 架构系统重构
* 规范与文档系统建设
* 提示词系统 / 知识系统设计
* 与 ChatGPT / Codex 协作的规范驱动开发

它不是“单纯的软件编码流程”，而是一个面向复杂系统的：

**目标驱动 + 规范驱动 + AI协作 + 状态桥接 + 分层演进 + 稳定基线迭代**

的开发闭环。

---

## 2. 核心原则

### 2.1 先定方向，再做实现

任何开发动作都不应直接从编码开始，而应先明确这次推进的主线、目标、边界与所属层级。

### 2.2 先规范化，再工程化

模糊想法必须先被收敛成可描述、可审视、可追踪的需求对象，再进入实现阶段。

### 2.3 变更必须受控

正式推进的改动应通过 OpenSpec 变更流程管理，而不是在无边界状态下直接修改代码或文档。

### 2.4 分层优先于堆功能

开发过程中应优先判断“它属于哪一层、是否破坏现有结构”，而不是只关心功能是否能跑。

### 2.5 状态需要显式化

系统开发不能仅依赖会话记忆或个人脑内状态，必须通过 digest 等状态桥机制维持项目连续性。

### 2.6 完成不等于写完

一个需求的结束，不是“代码已改完”，而是“已验证、已归档、已回到稳定基线”。

### 2.7 系统应以迭代方式演进

每一轮开发都应在稳定基线上发起下一轮，而不是长期在混乱的活动状态中叠加修改。

---

## 3. 总体流程

整个开发流程分为七个阶段：

1. 确定主线
2. 收敛需求
3. 发起规范变更
4. 协作实现
5. 验证与整合
6. 归档与回稳
7. 进入下一轮演进

其总体逻辑为：

**方向判定 → 需求收口 → OpenSpec 变更 → AI协作实现 → 验证整合 → 归档回稳 → 下一轮迭代**

---

## 4. 阶段一：确定主线

### 4.1 目标

明确当前最应该推进的系统主线，而不是直接进入局部实现。

### 4.2 需要回答的问题

* 现在到底在做什么
* 这是底层能力建设，还是上层应用建设
* 这是架构调整，还是功能补充
* 这是长期主线，还是短期修补
* 这一步是否会破坏已有结构
* 当前最优先的是“扩展能力”还是“稳住基线”

### 4.3 产物

* 当前主目标描述
* 当前主线判断
* 范围与边界说明
* 是否值得进入正式变更流程的判断

### 4.4 判定原则

若问题仍停留在方向不清、边界模糊、层级不明的状态，则不应直接进入实现。

---

## 5. 阶段二：收敛需求

### 5.1 目标

将原始想法从“讨论对象”转化为“规范对象”。

### 5.2 需要完成的事情

将需求明确为以下内容：

* 要解决什么问题
* 想达成什么结果
* 与现有系统的关系是什么
* 会影响哪些已有模块
* 哪些内容属于本次范围
* 哪些内容明确不做

### 5.3 常见需求类型

* 新功能
* 对现有功能的修改
* 架构重构
* 实现层优化
* 规范文档调整
* 系统分层重组
* 稳定性 / 可维护性建设

### 5.4 产物

形成一份可供 OpenSpec / Codex / ChatGPT 共同理解的功能描述或需求说明。

---

## 6. 阶段三：发起 OpenSpec 变更

### 6.1 目标

把一次开发活动纳入受控变更，而不是直接进入无记录修改。

### 6.2 标准动作

正式推进前，应完成以下动作：

* 查看当前 `digest.md` / `digest.local.md`
* 确认当前仓库主线和状态
* 确认是否已有 active change
* 选择新的 `change-id`
* 起草 proposal
* 补充 tasks
* 在必要时补充 design
* 对受影响功能编写增量 spec

### 6.3 作用

通过 OpenSpec，将本次开发活动变成：

* 可追踪的
* 可解释的
* 可协作的
* 可归档的
* 可持续演进的

### 6.4 原则

若一个改动已经影响系统能力、边界、契约、结构或可复用性，则应优先走 OpenSpec，而不是先改后补。

---

## 7. 阶段四：协作实现

### 7.1 目标

在规范明确之后，使用 AI 工具高效推进实现，但保持人对方向和边界的主导。

### 7.2 人机分工

在当前工作模式中，分工通常为：

* **我**：负责主线判断、边界定义、系统分层、需求约束、优先级选择
* **ChatGPT**：负责需求收敛、结构整理、架构分析、流程组织、Codex 输入整理
* **Codex**：负责按规范进行代码、文档、结构的具体实现

### 7.3 原则

AI 的角色是加速执行，不是替代架构判断。
任何实现都应服从已确定的：

* 目标
* 边界
* 分层
* 契约
* 规范

### 7.4 产物

* 代码变更
* 文档变更
* UI / 数据结构调整
* 架构抽象收口
* 规范配套补充

---

## 8. 阶段五：验证与整合

### 8.1 目标

确认改动不是局部可用，而是已经与整个系统正确接合。

### 8.2 核心检查项

* 构建是否通过
* 规范是否一致
* 是否符合 proposal / tasks / design 的目标
* 是否破坏原有层次结构
* 是否引入不必要耦合
* 上下层契约是否仍清晰
* 文档与实现是否同步

### 8.3 本阶段关注点

不是“它能不能运行”，而是：

* 是否融入现有系统
* 是否保持结构清晰
* 是否能作为后续迭代基线继续存在

---

## 9. 阶段六：归档与回稳

### 9.1 目标

让一次开发活动真正结束，并恢复仓库到可持续推进的稳定状态。

### 9.2 标准动作

完成以下事项：

* 确认 change 已完成
* archive 当前 change
* 更新 digest / digest.local
* 同步当前基线状态
* 确认仓库无悬空主线
* 回到“可发起下一轮变更”的干净状态

### 9.3 核心认识

完成一个需求，不是“做完了”，而是：

**已经归档，且系统恢复为稳定基线。**

---

## 10. 阶段七：进入下一轮演进

### 10.1 目标

基于稳定基线重新判断下一步，而不是在旧变更余波中被动推进。

### 10.2 需要回答的问题

* 当前最值得推进的下一条主线是什么
* 该补能力、补产品、补架构还是补稳定性
* 是否需要先稳住已有成果
* 当前阶段的演进重点是什么

### 10.3 结果

形成下一轮“主线选择 → 需求收口 → 发起变更”的新循环。

---

## 11. 分层原则

这套流程不是单纯按功能推进，而是强调先判断层次，再决定如何开发。

### 11.1 开发时必须优先判断

* 这部分属于核心原则层，还是应用层
* 是底层能力，还是场景编排
* 是架构约束，还是具体实例
* 是否误把上层场景写进了底层核心

### 11.2 原则

底层应尽可能保持抽象、稳定、可复用；
上层可以服务具体场景，但不应反向污染底层结构。

### 11.3 典型表现

例如在你的系统中，经常要区分：

* Core / Principles
* Architecture / Contracts
* Terms / Orchestration
* Profiles / 场景模板
* App / UI / 具体应用层

这说明分层判断已经是开发流程的一部分，而不是事后整理动作。

---

## 12. digest 的角色定义

### 12.1 定位

`digest.md` / `digest.local.md` 是全局状态桥，不是日志。

### 12.2 它的作用

用于显式记录：

* 当前主目标
* 当前阶段
* 当前主线
* 仓库状态
* 已归档/活动变更状态
* 阻塞项
* 下一步建议

### 12.3 它连接的对象

digest 连接了：

* 人的当前理解
* ChatGPT 的上下文入口
* Codex 的协作背景
* Git 当前基线
* OpenSpec 当前变更状态

### 12.4 原则

开发连续性必须依赖显式状态桥，而不能只依赖“上一轮对话还记得”。

---

## 13. AI 协作规范

### 13.1 ChatGPT 的适用职责

适合承担：

* 主线整理
* 架构分析
* 分层判断
* 需求收敛
* OpenSpec 需求草案整理
* 给 Codex 的实现输入整理
* 阶段性总结与下一步建议

### 13.2 Codex 的适用职责

适合承担：

* 按需求生成代码
* 修改目录结构
* 调整文档
* 执行规范流程中的实现任务
* 根据 proposal / tasks / design 完成变更

### 13.3 原则

AI 协作必须建立在显式目标、显式边界、显式上下文之上。
不能把未收敛的问题直接丢给执行器。

---

## 14. 完整闭环定义

一个完整、健康的开发闭环应满足：

1. 有明确主线
2. 有清晰需求边界
3. 有 OpenSpec 变更记录
4. 有可解释实现
5. 有验证与整合
6. 有归档与 digest 同步
7. 有稳定基线
8. 有下一轮演进入口

只满足前半段，不算真正完成。

---

## 15. 一句话版流程定义

我的系统开发流程是：

**先确定主线和边界，再将需求规范化，通过 OpenSpec 发起受控变更，借助 AI 工具完成实现，并通过 digest 维持全局状态一致，在验证、归档并恢复稳定基线后，进入下一轮系统演进。**

---

## 16. 简版执行清单

每次开发时，按下面顺序检查：

### A. 先判断

* 现在要推进的主线是什么
* 它属于哪一层
* 这次是扩能力、改架构、补功能还是稳基线

### B. 再收口

* 本次目标是什么
* 边界是什么
* 不做什么
* 影响哪些已有部分

### C. 进入规范

* 看 digest
* 确认 active change
* 定 change-id
* 起草 proposal / tasks / design / spec

### D. 推进实现

* 给 Codex 明确需求
* 按规范落地代码和文档
* 保持结构一致

### E. 做整合

* 检查构建
* 检查契约
* 检查文档一致性
* 检查分层是否被破坏

### F. 结束变更

* archive
* 更新 digest
* 回到稳定基线

### G. 进入下一轮

* 总结当前成果
* 判断下一条主线
* 发起新的受控演进

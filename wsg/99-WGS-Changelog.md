# WGS Changelog

## 1. 本文件的定位

WGS 不是一个一开始就会完全定型的系统。
它更像一套会随着实际使用不断收束、修订、扩展的母规范。

因此，WGS 需要一份专门的变更记录文件，用来回答：

* WGS 在不同版本中改了什么
* 为什么要改
* 哪些模块或规则发生了调整
* 当前版本相对上一版最重要的变化是什么

这就是 Changelog 的作用。

Changelog 不是用来记录某个具体世界的演化历史。
它记录的是：

**WGS 这套系统本身的演化历史。**

也就是说，本文件记录的是：

* 模块顺序是否改动
* 状态卡字段是否调整
* 工作流是否重组
* 调用协议是否收束
* Prompt Profiles 是否更新
* 文档结构是否变化

它记录的是“系统怎么变了”，而不是“世界里发生了什么”。

---

## 2. Changelog 的作用

WGS 的 Changelog 主要有以下几个作用。

### 2.1 记录系统收束过程

WGS 是从实践中逐步收束出来的。
Changelog 可以保留这种收束痕迹，避免未来忘记“为什么会这样设计”。

### 2.2 方便回看关键决策

比如：

* 为什么确定八模块结构
* 为什么要把时代放在个体前面
* 为什么强调状态卡
* 为什么要把叙事层与客观层分开

这些关键决策，都值得在 Changelog 中留痕。

### 2.3 帮助后续迭代更稳定

如果以后你继续修改 WGS，Changelog 可以帮助你判断：

* 这次改动是局部优化，还是整体方向变化
* 新改动是否推翻了旧的核心原则
* 文档之间是否需要同步更新

### 2.4 为未来版本提供脉络

当 WGS 从 v0.1 长到 v0.2、v0.3 时，Changelog 会成为最清晰的版本脉络图。

---

## 3. 记录原则

WGS 的 Changelog 建议遵循以下原则。

### 3.1 记录系统级变化，不记录普通日常对话

不是每一次聊天都要写入 Changelog。
只有那些真正影响 WGS 结构、原则、模块、调用方式的变化，才值得记录。

### 3.2 优先记录“为什么改”

仅仅写“改了什么”还不够。
更重要的是说明：

* 为什么要改
* 这个改动解决了什么问题
* 会影响哪些部分

### 3.3 记录应尽量压缩

Changelog 不是重复写文档正文。
它应该尽量简洁，突出版本变化点和影响范围。

### 3.4 保持版本视角

每条记录应尽量能回答：

* 这是哪个版本的变化
* 变化相对于上一版意味着什么

---

## 4. 推荐记录格式

WGS 的 Changelog 建议采用如下格式：

```text id="wgs-changelog-format"
## [版本号] - 日期

### 新增
-

### 调整
-

### 移除
-

### 原因
-

### 影响范围
-
```

如果某个版本没有“移除”项，可以省略。
如果某次变化很小，也可以只写“调整 + 原因 + 影响范围”。

---

## 5. 推荐版本标记方式

WGS 目前建议采用简洁版本号，例如：

* v0.1
* v0.2
* v0.3

如果后续需要更细，可以使用：

* v0.1.1
* v0.1.2
* v0.2.0

但在早期阶段，保持简单更重要。
建议先不要把版本号体系搞得太复杂。

## 6. v0.1 初始记录（建议首条）

下面是一条适合作为 WGS 初始版本记录的示例。

---

## [v0.1] - 初始骨架建立

### 新增

* 建立 WGS 作为纯提示词驱动的世界生成系统的核心定义。
* 明确 WGS 不是一次性世界观写作模板，而是状态驱动、可迭代演化的系统。
* 建立八模块主架构：混沌、盘古、女娲、伏羲、羲和、神农、共工、庄周。
* 明确三大层级：世界本体层、世界演化层、世界投影层。
* 引入世界状态卡，作为多轮迭代的公共承接底座。
* 建立三大工作流：创世、演化、投影。
* 建立统一调用协议与模块 Prompt Profiles。
* 建立场景矩阵与使用说明文档骨架。

### 调整

* 将模块顺序正式确定为：混沌 → 盘古 → 女娲 → 伏羲 → 羲和 → 神农 → 共工 → 庄周。
* 确定“时代生成器”位于“个体生成器”之前，以保证个体生成具备明确的历史空气与时代约束。
* 确定庄周只负责叙事投影，不负责补写客观世界根基。

### 原因

* 需要将原本较模糊的“世界生成提示词想法”收束为一套正式母规范。
* 需要解决世界生成过程中“层级混杂、状态不承接、叙事与客观层混淆”的问题。
* 需要让系统具备长期迭代能力，而不是停留在一次性 prompt 生成。

### 影响范围

* 影响 WGS 全部主文档结构。
* 影响后续 Prompt Profiles、使用方式与测试世界构建方法。
* 为未来版本的收束和扩展提供基础骨架。

## 7. 后续适合记录的变化类型

以后当你继续调整 WGS 时，以下变化通常值得写进 Changelog。

### 7.1 架构变化

例如：

* 模块顺序改动
* 新增或删除模块
* 三层结构调整
* 工作流重构

### 7.2 协议变化

例如：

* 调用模板结构改动
* 标准输出格式改动
* 任务类型枚举调整

### 7.3 状态卡变化

例如：

* 增加新字段
* 调整稳定层与活跃层写法
* 引入更压缩的简版状态卡

### 7.4 Prompt Profile 变化

例如：

* 重写某模块 profile
* 拆出某模块的多个变体
* 调整某模块的默认约束

### 7.5 使用方法变化

例如：

* 发现最优的新手路径
* 调整推荐停驻点
* 增加新的常见错误提示

---

## 8. 不适合写进 Changelog 的内容

为了避免 Changelog 失焦，以下内容通常不必写进去。

### 8.1 某个具体世界的设定细节

这些应写进该世界自己的状态卡，而不是 WGS 的 Changelog。

### 8.2 单次普通调用结果

比如某次用混沌生成了什么，这不属于系统级变化。

### 8.3 纯措辞微调

如果只是改了一两句话的表述，但不影响系统结构和使用方式，可以不单独记录。

### 8.4 重复性整理

例如只是在不同文件里同步同一改动，没有新增系统含义，也不一定需要单独写。

---

## 9. Changelog 的更新时机

WGS 的 Changelog 不需要每次聊天都更新。
更合适的时机通常有：

### 9.1 一个阶段性骨架定下来之后

例如：

* 八模块结构确定
* 工作流定型
* 状态卡结构收束

### 9.2 一次明显的系统方向调整之后

例如：

* 从“世界版 SLS”转向“独立的迭代式世界生成系统”
* 从一次性生成思路转向状态驱动思路
* 从模块并列思路转向主顺序 + 可回调结构

### 9.3 完成一轮真实测试并反推修改之后

当你拿一个测试世界实际跑完一遍后，往往会得到最有价值的系统级反馈，这很适合进入 Changelog。

---

## 10. Changelog 的最简理解

可以把本文件最简地理解为：

**WGS 这套系统自己的版本演化记录。**

它记录的不是世界的故事，
而是：

* WGS 为什么会长成现在这样
* WGS 将来又是如何继续变化的

---

## 11. 当前阶段建议

在完成 v0.1 文档骨架之后，下一步最适合的事情通常不是继续抽象系统本身，而是：

1. 选一个测试世界
2. 按最小路径跑一遍 WGS
3. 记录哪些地方顺、哪些地方卡
4. 再基于真实使用结果更新 Changelog

也就是说，Changelog 最有价值的后续内容，会来自真实运行，而不只是继续理论推演。
